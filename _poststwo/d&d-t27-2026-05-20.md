# 🎯 **Design Web Crawler**


## 1️⃣ Core Framework

When discussing **Web Crawler design**, I frame it as:

1. URL discovery and URL frontier
2. Crawl scheduling and prioritization
3. Robots.txt and politeness
4. Fetching and retry
5. Parsing and link extraction
6. Deduplication and canonicalization
7. Storage and indexing pipeline
8. Scaling, freshness, and failure handling

---

## 2️⃣ Core Requirements

---

### Functional Requirements

* Crawl web pages from seed URLs
* Discover new URLs from pages
* Respect robots.txt
* Respect per-domain rate limits
* Fetch HTML and other supported content
* Parse pages and extract links
* Deduplicate URLs and content
* Store crawled pages
* Support recrawling for freshness
* Support crawl priority

---

### Non-functional Requirements

* High scalability
* High throughput
* Fault tolerant
* Polite to websites
* Avoid duplicate crawling
* Efficient storage
* Configurable crawl frequency
* Eventually consistent crawl index is acceptable

---

**👉 Interview Answer**

> A web crawler starts from seed URLs,
> fetches pages,
> parses content,
> extracts links,
> deduplicates URLs,
> and schedules new URLs for future crawling.
>
> The main challenges are scalability,
> politeness,
> deduplication,
> freshness,
> and avoiding crawling the same or low-value content repeatedly.

---

---

```http

```

Request:

```json
{
  "urls": [
    "https://example.com",
    "https://news.example.com"
  ],
  "priority": "high"

```

---

### Get Crawl Status

```http
GET /api/crawl/status?url=https://example.com
```

---

### Submit Crawl Job

```http
POST /api/crawl/jobs
```

Request:

```json
{
  "domain": "example.com",
  "maxDepth": 3,
  "allowedContentTypes": ["text/html"],
  "recrawlIntervalHours": 24
}
```

---

### Fetch Crawled Page

```http
GET /api/pages?url=https://example.com/page1
```

---

**👉 Interview Answer**

> The system may expose APIs to add seed URLs,
> configure crawl jobs,
> check crawl status,
> and retrieve crawled pages.
>
> Internally, the most important interface is the URL frontier,
> which decides what URL should be fetched next.

---

```text
Seed URLs
→ URL Frontier
→ Crawl Scheduler
→ Robots / Politeness Service
→ Fetcher Workers
→ Parser / Link Extractor
→ URL Canonicalizer
→ Dedup Service
→ Page Storage
→ Indexing Pipeline
→ Recrawl Scheduler
```

---

## Main Components

### URL Frontier

* Stores URLs waiting to be crawled
* Tracks priority
* Tracks next eligible crawl time

---

### Crawl Scheduler

* Selects next URLs to fetch
* Enforces priority and politeness

---

### Fetcher Workers

* Download web pages
* Handle timeout and retry
* Record HTTP status

---

### Parser

* Extracts text, metadata, and links
* Detects canonical URL
* Identifies page language and content type

---

### Dedup Service

* Deduplicates URLs
* Deduplicates page content

---

### Storage

* Stores raw HTML
* Stores extracted text
* Stores metadata
* Stores link graph

---

**👉 Interview Answer**

> I would separate crawling into URL frontier,
> scheduler,
> fetchers,
> parsers,
> deduplication,
> storage,
> and indexing.
>
> This separation allows the system to scale each stage independently
> and makes failures easier to isolate.

# 6️⃣ URL Frontier

---

## What Is URL Frontier?

URL frontier is the queue of URLs waiting to be crawled.

It decides:

```text
what to crawl next
when to crawl it
with what priority
```

---

## Frontier Requirements

* Prioritize important URLs
* Avoid duplicate URLs
* Respect domain politeness
* Support recrawling
* Support distributed workers
* Support retry scheduling

---

## Priority Signals

* Page rank / link popularity
* Domain importance
* Content freshness
* Crawl depth
* Business rules
* User-submitted priority
* Historical update frequency

---

**👉 Interview Answer**

> URL frontier is the heart of a crawler.
>
> It stores URLs waiting to be crawled,
> assigns priority,
> respects per-domain crawl delay,
> and schedules recrawls.
>
> A good frontier prevents duplicate work
> and ensures important pages are crawled first.

---

# 7️⃣ URL Canonicalization

---

## Why Needed?

The same page can have many URL forms.

Examples:

```text
https://example.com
https://example.com/
https://www.example.com/
https://example.com?utm_source=abc
```

---

## Canonicalization Rules

* Lowercase scheme and host
* Remove default ports
* Normalize trailing slash
* Remove tracking parameters
* Sort query parameters
* Resolve redirects
* Respect canonical link tag
* Normalize percent encoding

---

**👉 Interview Answer**

> URL canonicalization reduces duplicate crawling.
>
> Before adding a URL to the frontier,
> I would normalize it by lowercasing host,
> removing tracking parameters,
> resolving redirects,
> and applying canonical URL rules when available.

---

# 8️⃣ Robots.txt and Politeness

---

## Robots.txt

Before crawling a domain:

```text
https://example.com/robots.txt
```

The crawler must check:

* Allowed paths
* Disallowed paths
* Crawl delay
* Sitemap location

---

## Politeness

Do not overload websites.

Rules:

* Limit requests per domain
* Add delay between requests
* Respect crawl-delay
* Use backoff on errors
* Avoid aggressive parallel fetches to same host

---

**👉 Interview Answer**

> A crawler must respect robots.txt and politeness rules.
>
> Before fetching URLs from a domain,
> the crawler should load robots.txt,
> check whether the path is allowed,
> and enforce domain-level crawl delay.
>
> This prevents harming target websites.

---

# 9️⃣ Fetcher Design

---

## Fetch Flow

```text
Scheduler assigns URL
→ Fetcher checks robots and politeness
→ Fetcher sends HTTP request
→ Follow redirects within limit
→ Validate content type and size
→ Store response metadata
→ Send content to parser
```

---

## Fetcher Responsibilities

* Timeout handling
* Retry temporary failures
* Limit max page size
* Follow redirects safely
* Validate content type
* Record HTTP status
* Avoid infinite redirect loops
* Use compression support

---

## Retry Rules

```text
5xx / timeout → retry with backoff
404 / 410 → mark not found
403 → mark forbidden
429 → backoff domain
```

---

**👉 Interview Answer**

> Fetcher workers download pages and handle HTTP behavior.
>
> They should enforce timeout,
> max size,
> redirect limits,
> content type checks,
> and retry rules.
>
> Temporary failures can be retried,
> but permanent failures should update crawl state.

---

# 🔟 Parser and Link Extraction

---

## Parser Responsibilities

* Extract title
* Extract main text
* Extract links
* Extract metadata
* Detect language
* Detect canonical URL
* Extract structured data
* Remove boilerplate if needed

---

## Link Extraction

From:

```html
<a href="/page2">Next Page</a>
```

Convert to absolute URL:

```text
https://example.com/page2
```

---

## Output

```text
parsed content
new discovered URLs
metadata
content hash
```

---

**👉 Interview Answer**

> The parser extracts content and links from fetched pages.
>
> It converts relative links to absolute URLs,
> canonicalizes discovered URLs,
> and sends new URLs back to the frontier.
>
> It also extracts text and metadata for indexing.

# 1️⃣1️⃣ Deduplication

---

## URL Deduplication

Avoid crawling same URL repeatedly.

Use:

```text
normalized_url hash
```

---

## Content Deduplication

Different URLs may serve same content.

Use:

```text
content hash
simhash
minhash
```

---

## Exact Duplicate

Same content hash.

---

## Near Duplicate

Very similar pages.

Examples:

* Printer-friendly pages
* Tracking parameter variants
* Duplicate category pages
* Mirror sites

---

**👉 Interview Answer**

> Deduplication happens at both URL and content level.
>
> URL deduplication avoids scheduling the same normalized URL.
>
> Content deduplication detects duplicate or near-duplicate pages
> using hashes such as content hash or simhash.

---

## Raw HTML Storage


---

## Extracted Content Storage

Store:

* Text

Store links between pages:

## Search Index

Parsed content can feed:

```text
search index

ML pipeline
```

---

**👉 Interview Answer**

> I would store raw HTML in object storage,
> extracted text and metadata in a database or document store,
> and link relationships in a link graph store.
>
> Search indexing should be asynchronous,
> driven by page-crawled events.

# 1️⃣3️⃣ Recrawling and Freshness

---

## Why Recrawl?

Pages change over time.

Examples:

* News pages update frequently
* Product pages change price
* Static docs rarely change

---

## Recrawl Frequency Signals

* Historical change frequency
* Page importance
* Content type
* HTTP cache headers
* Sitemap hints
* Last modified timestamp
* Business priority

---

## Recrawl Flow

```text
Page crawled
→ Estimate next_crawl_at
→ Add URL back to frontier
→ Recrawl when due
```

---

**👉 Interview Answer**

> Recrawling is needed to keep content fresh.
>
> I would schedule recrawl frequency based on page importance,
> historical change rate,
> HTTP cache headers,
> sitemap hints,
> and business priority.
>
> Not every page should be recrawled at the same frequency.

---

# 1️⃣4️⃣ Sitemaps

---

## Why Useful?

Sitemaps tell crawlers important URLs.

Example:

```text
https://example.com/sitemap.xml
```

---

## Benefits

* Discover pages not linked from homepage
* Understand update frequency
* Find canonical URLs
* Prioritize important pages

---

## Flow

```text
Discover sitemap from robots.txt
→ Fetch sitemap
→ Extract URLs
→ Add URLs to frontier
```

---

**👉 Interview Answer**

> Sitemaps are useful seed sources.
>
> The crawler can discover sitemap URLs from robots.txt,
> parse them,
> and add listed URLs to the frontier with appropriate priority.

---

# 1️⃣5️⃣ Distributed Crawling

---

## Why Distributed?

A single machine cannot crawl the web.

---

## Partitioning Strategies

### By Domain

```text
hash(domain) % N
```

Pros:

* Easier politeness control

---

### By URL Hash

```text
hash(normalized_url) % N
```

Pros:

* Better load distribution

---

### Hybrid

```text
domain-level scheduling + URL-level workers
```

---

## Important Rule

Requests to the same domain should be coordinated.

---

**👉 Interview Answer**

> Distributed crawling requires partitioning.
>
> I would partition scheduling by domain to enforce politeness,
> while fetcher workers can scale horizontally.
>
> The system must coordinate requests to the same domain
> so it does not overload websites.

---

# 1️⃣6️⃣ Priority and Scheduling

---

## Scheduling Goals

* Crawl important pages first
* Keep fresh content updated
* Avoid low-value duplicate pages
* Balance domains fairly
* Respect crawl budget

---

## Priority Queue

Each URL has:

```text
priority_score
next_crawl_at
domain
```

---

## Scheduling Flow

```text
Pick eligible domains
→ Pick highest priority URL per domain
→ Enforce crawl delay
→ Assign to fetcher
```

---

**👉 Interview Answer**

> The scheduler should not simply crawl FIFO.
>
> It should prioritize URLs by importance,
> freshness needs,
> crawl depth,
> domain quality,
> and business rules,
> while enforcing per-domain crawl delay.

# 1️⃣7️⃣ Failure Handling

---

## Common Failures

* Fetch timeout
* DNS failure
* 5xx response
* 404 / 410 removed page
* 429 rate limited
* Parser failure
* Duplicate URL explosion
* Malformed HTML
* Infinite crawl traps
* Worker crash

---

## Strategies

* Retry with backoff
* Per-domain error budget
* Mark permanent failures
* Dead-letter queue
* Crawl trap detection
* URL pattern blacklist
* Worker checkpointing
* Replay crawl tasks

---

**👉 Interview Answer**

> Crawlers must handle many failures.
>
> Temporary failures should be retried with backoff,
> permanent failures should update URL status,
> and repeated domain failures should reduce crawl rate.
>
> Crawl traps and URL explosions should be detected and limited.

---

# 1️⃣8️⃣ Crawl Traps

---

## What Is a Crawl Trap?

A site generates infinite or near-infinite URLs.

Examples:

```text
/calendar/2026/05/03
/calendar/2026/05/04
/search?q=a
/search?q=aa
/session?id=123
```

---

## Solutions

* Limit crawl depth
* Limit URLs per domain
* Detect repeating URL patterns
* Remove session IDs
* Block infinite query parameters
* Use content deduplication
* Prefer canonical URLs

---

**👉 Interview Answer**

> Crawl traps can waste huge crawl resources.
>
> I would limit crawl depth,
> cap URLs per domain,
> detect repeating URL patterns,
> remove session parameters,
> and use content deduplication to avoid infinite crawling.

---

# 1️⃣9️⃣ Consistency Model

---

## Stronger Consistency Needed For

* URL deduplication
* Robots.txt enforcement
* Domain politeness
* Crawl task assignment
* Access control for stored pages

---

## Eventual Consistency Acceptable For

* Search indexing
* Page metadata updates
* Link graph updates
* Recrawl scheduling
* Analytics

---

**👉 Interview Answer**

> Web crawling mostly accepts eventual consistency.
>
> Search indexing and metadata updates can lag behind crawling.
>
> But URL deduplication,
> robots.txt enforcement,
> and politeness rules need stronger correctness
> to avoid duplicate work or violating crawl policies.

# 2️⃣1️⃣ End-to-End Flow

---

## Crawl Flow

```text
Seed URL added
→ URL canonicalized
→ URL inserted into frontier
→ Scheduler picks eligible URL
→ Robots and politeness checked
→ Fetcher downloads page
→ Parser extracts content and links
→ Dedup service filters URLs
→ New URLs go back to frontier
→ Page stored and indexed
```

---

## Recrawl Flow

```text
Page crawled
→ Freshness policy calculates next_crawl_at
→ URL reinserted into frontier
→ Scheduler recrawls when due
```

---

## Failure Flow

```text
Fetch fails
→ Classify temporary vs permanent
→ Retry with backoff or mark failed
→ Update URL status
→ Adjust domain crawl rate if needed
```

---

## Key Insight

> Web Crawler is not just downloading pages —
> it is a large-scale scheduling, politeness, deduplication, and freshness system.

# 🧠 Staff-Level Answer (Final)

---

**👉 Interview Answer (Full Version)**

> When designing a web crawler,
> I think of it as a large-scale URL scheduling and content ingestion system.
>
> The crawler starts from seed URLs,
> canonicalizes them,
> inserts them into a URL frontier,
> and schedules fetch tasks based on priority,
> crawl freshness,
> and domain politeness rules.
>
> Before crawling a domain,
> the system should fetch and respect robots.txt,
> including allowed paths,
> disallowed paths,
> crawl delay,
> and sitemap hints.
>
> The URL frontier is the core component.
> It tracks what URLs are waiting to be crawled,
> when they are eligible,
> and how important they are.
>
> Fetcher workers download pages,
> enforce timeout,
> redirect limits,
> content type checks,
> and retry rules.
>
> After fetching,
> parsers extract text,
> metadata,
> canonical URL,
> and outgoing links.
>
> New links are canonicalized,
> deduplicated,
> and added back into the frontier.
>
> Deduplication happens at both URL level and content level.
> URL normalization avoids crawling the same URL repeatedly,
> while content hashes or simhash detect duplicate or near-duplicate pages.
>
> For storage,
> I would keep raw HTML in object storage,
> extracted content and metadata in a database or document store,
> and link relationships in a link graph.
>
> Recrawling is needed for freshness.
> The next crawl time should depend on page importance,
> historical update frequency,
> cache headers,
> sitemap hints,
> and business priority.
>
> To scale,
> I would distribute the crawler by domain or URL hash,
> but still coordinate requests per domain
> to enforce politeness.
>
> The main trade-offs are crawl throughput,
> freshness,
> politeness,
> storage cost,
> duplicate avoidance,
> and crawl coverage.
>
> Ultimately, the goal is to discover and ingest high-value pages efficiently,
> keep them fresh,
> and avoid harming websites or wasting resources on duplicates and crawl traps.

# 6️⃣ URL Frontier

---

## 什么是 URL Frontier？

URL frontier 是等待抓取的 URL 队列。

它决定：

```text
what to crawl next
when to crawl it
with what priority
```

---

## Frontier Requirements

* 优先抓取重要 URLs
* 避免重复 URLs
* 遵守 domain politeness
* 支持 recrawling
* 支持 distributed workers
* 支持 retry scheduling

---

## Priority Signals

* Page rank / link popularity
* Domain importance
* Content freshness
* Crawl depth
* Business rules
* User-submitted priority
* Historical update frequency

---

**👉 面试回答**

> URL frontier 是 crawler 的核心。
>
> 它存储等待抓取的 URLs，
> 分配 priority，
> 遵守 per-domain crawl delay，
> 并调度 recrawls。
>
> 一个好的 frontier 可以避免重复工作，
> 并确保重要页面优先被抓取。

# 7️⃣ URL Canonicalization

---

## 为什么需要？

同一个页面可能有很多 URL 形式。

示例：

```text
https://example.com
https://example.com/
https://www.example.com/
https://example.com?utm_source=abc
```

---

## Canonicalization Rules

* Lowercase scheme and host
* Remove default ports
* Normalize trailing slash
* Remove tracking parameters
* Sort query parameters
* Resolve redirects
* Respect canonical link tag
* Normalize percent encoding

---

**👉 面试回答**

> URL canonicalization 可以减少重复抓取。
>
> 在将 URL 加入 frontier 前，
> 我会对它进行 normalization，
> 例如 host 小写、移除 tracking parameters、
> 解析 redirects，
> 并在页面提供 canonical URL 时使用它。

# 8️⃣ Robots.txt and Politeness

---

## Robots.txt

抓取 domain 前：

```text
https://example.com/robots.txt
```

Crawler 必须检查：

* Allowed paths
* Disallowed paths
* Crawl delay
* Sitemap location

---

## Politeness

不要对网站造成压力。

规则：

* Limit requests per domain
* Add delay between requests
* Respect crawl-delay
* Use backoff on errors
* Avoid aggressive parallel fetches to same host

---

**👉 面试回答**

> Crawler 必须遵守 robots.txt 和 politeness rules。
>
> 在抓取一个 domain 的 URLs 之前，
> crawler 应该加载 robots.txt，
> 检查 path 是否允许抓取，
> 并执行 domain-level crawl delay。
>
> 这样可以避免伤害目标网站。
