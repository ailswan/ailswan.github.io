## PPT 1 — Title Slide

Hello everyone.
Today I’d like to walk through another important dynamic programming pattern: **Sequence DP**.

Sequence DP is used when the input has a natural order, usually an array or a string.

The basic idea is:

```text
Use previous positions
→ Compute the current position
→ Build the answer from left to right
```

This pattern appears in Maximum Subarray, Longest Increasing Subsequence, Word Break, Decode Ways, House Robber, Min Cost Climbing Stairs, Paint House, and many string or array DP problems.

The key idea is:

**Sequence DP connects the current position with previous positions.**

## PPT 2 — Core Framework

When I think about Sequence DP, I usually break it down into five parts.

First, what problem pattern it solves.
Second, how to define `dp[i]`.
Third, prefix-based DP.
Fourth, ending-at-index DP.
And finally, common examples and senior-level explanation.

The most important question is:

```text
What does dp[i] mean?
```

There are two common meanings.

The first is:

```text
dp[i] = answer using the first i elements
```

The second is:

```text
dp[i] = best answer ending exactly at index i
```

These two meanings lead to very different recurrences.

The key idea is:

**Before writing recurrence, define the state in plain English.**

## PPT 3 — Prefix DP vs Ending-at-Index DP

Prefix DP is useful when we are solving a problem about a prefix.

For example:

```text
dp[i] = whether s[0:i] can be segmented
```

This is common in Word Break, Decode Ways, and Min Cost Climbing Stairs.

Ending-at-index DP is different.

It means the current element must be included.

For example:

```text
dp[i] = maximum subarray sum ending at index i
```

or:

```text
dp[i] = LIS length ending at index i
```

This is common in Maximum Subarray, Longest Increasing Subsequence, Arithmetic Slices, and continuous sequence problems.

The key idea is:

**Prefix DP asks what is true for the prefix. Ending DP asks what ends here.**

**(PPT: Prefix State vs Ending State)**

## PPT 4 — Maximum Subarray

A classic ending-at-index problem is **Maximum Subarray**.

The problem asks for the contiguous subarray with the largest sum.

State definition:

```text
dp[i] = maximum subarray sum ending exactly at index i
```

At index `i`, we have two choices:

```text
start a new subarray at nums[i]
or
extend the previous subarray
```

So the recurrence is:

```text
dp[i] = max(nums[i], dp[i - 1] + nums[i])
```

Code:

```python
def max_sub_array(nums):
    current = nums[0]
    best = nums[0]

    for i in range(1, len(nums)):
        current = max(nums[i], current + nums[i])
        best = max(best, current)

    return best
```

The key idea is:

**For ending-at-index DP, the global answer is often max over all states.**

**(PPT: Maximum Subarray — Kadane’s Algorithm)**

## PPT 5 — Longest Increasing Subsequence

Another classic ending-at-index problem is **Longest Increasing Subsequence**.

State definition:

```text
dp[i] = length of the longest increasing subsequence ending exactly at index i
```

To compute `dp[i]`, we check every previous index `j`.

If:

```text
nums[j] < nums[i]
```

then `nums[i]` can extend the subsequence ending at `j`.

So:

```text
dp[i] = max(dp[i], dp[j] + 1)
```

Code:

```python
def length_of_lis(nums):
    n = len(nums)
    dp = [1] * n

    for i in range(n):
        for j in range(i):
            if nums[j] < nums[i]:
                dp[i] = max(dp[i], dp[j] + 1)

    return max(dp)
```

The key idea is:

**LIS compares the current element with all previous candidates.**

**(PPT: LIS — Ending-at-Index DP)**

## PPT 6 — LIS Binary Search Optimization

LIS also has an optimized solution using binary search.

We maintain an array called `tails`.

```text
tails[length - 1] = smallest possible ending value
for an increasing subsequence of that length
```

Code:

```python
import bisect

def length_of_lis(nums):
    tails = []

    for num in nums:
        index = bisect.bisect_left(tails, num)

        if index == len(tails):
            tails.append(num)
        else:
            tails[index] = num

    return len(tails)
```

This does not store the actual subsequence.

But it stores enough information to compute the length.

The complexity improves from **O(n²)** to **O(n log n)**.

The key idea is:

**Smaller tails leave more room for future extension.**

**(PPT: LIS Optimized — Binary Search + Tails)**

## PPT 7 — Word Break

Now let’s look at prefix DP.

A classic example is **Word Break**.

The problem asks whether a string can be segmented into dictionary words.

State definition:

```text
dp[i] = whether s[0:i] can be segmented
```

To compute `dp[i]`, we try every split point `j`.

If:

```text
dp[j] is true
and
s[j:i] is in the dictionary
```

then `dp[i]` is true.

Code:

```python
def word_break(s, word_dict):
    word_set = set(word_dict)
    n = len(s)

    dp = [False] * (n + 1)
    dp[0] = True

    for i in range(1, n + 1):
        for j in range(i):
            if dp[j] and s[j:i] in word_set:
                dp[i] = True
                break

    return dp[n]
```

The key idea is:

**Prefix DP often tries split points.**

**(PPT: Word Break — Prefix DP)**

## PPT 8 — Decode Ways

Another prefix DP example is **Decode Ways**.

We are given a digit string and need to count how many ways it can be decoded.

State definition:

```text
dp[i] = number of ways to decode s[0:i]
```

At each position, we check one digit and two digits.

If the last one digit is valid:

```text
dp[i] += dp[i - 1]
```

If the last two digits are between 10 and 26:

```text
dp[i] += dp[i - 2]
```

Code:

```python
def num_decodings(s):
    if not s:
        return 0

    n = len(s)
    dp = [0] * (n + 1)

    dp[0] = 1
    dp[1] = 1 if s[0] != "0" else 0

    for i in range(2, n + 1):
        one = s[i - 1]
        two = s[i - 2:i]

        if one != "0":
            dp[i] += dp[i - 1]

        if 10 <= int(two) <= 26:
            dp[i] += dp[i - 2]

    return dp[n]
```

The key idea is:

**Decode Ways counts valid transitions into the current prefix.**

**(PPT: Decode Ways — One Digit or Two Digits)**

## PPT 9 — Common Mistakes and Complexity

A common mistake in sequence DP is unclear state meaning.

For example, confusing:

```text
dp[i] = answer ending at i
```

with:

```text
dp[i] = answer using first i elements
```

Another common mistake is returning the wrong final answer.

For ending-at-index DP, the final answer is often:

```text
max(dp)
```

not necessarily:

```text
dp[-1]
```

This is true for Maximum Subarray and LIS.

For prefix DP, the final answer is often:

```text
dp[n]
```

Complexity can be analyzed as:

```text
number of states × transition cost
```

Maximum Subarray is **O(n)** because each state is O(1).

Basic LIS is **O(n²)** because each state scans previous positions.

The key idea is:

**The state definition determines both recurrence and final answer.**

**(PPT: Mistakes + Complexity)**

## PPT 10 — Senior-Level Explanation and Closing Insight

At senior level, I would explain Sequence DP like this.

Sequence DP applies when the input is ordered, and the answer at the current position depends on previous positions.

The first question is what `dp[i]` means.

There are two common choices.

Prefix-based state means `dp[i]` represents the answer for the first `i` elements, or the prefix `s[0:i]`.

Ending-at-index state means `dp[i]` represents the best answer that must end at index `i`.

For Maximum Subarray, the state ends at `i`.

For LIS, the state also ends at `i`, and we compare against previous indexes.

For Word Break and Decode Ways, the state represents a prefix.

The final answer may be `dp[n]` for prefix DP, or `max(dp)` for ending-at-index DP.

The final insight is:

**Sequence DP is not just writing a dp array.
It is choosing the correct meaning of position.**

If you can explain whether `dp[i]` is prefix-based or ending-at-index, how the current element connects to previous states, what the base case is, and why the final answer is `dp[n]` or `max(dp)`, then you understand Sequence DP at an interview-ready level.

Thank you.

**(PPT: Closing Insight — State Meaning Drives Recurrence)**
