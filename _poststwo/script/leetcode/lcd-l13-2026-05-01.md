## PPT 1 — Title Slide

Hello everyone.
Today I’d like to walk through a classic dynamic programming pattern: **Knapsack DP**.

Knapsack DP is used when we need to choose items under a constraint.

The basic idea is:

```text
For each item,
decide whether to take it or skip it,
while respecting a capacity or target constraint.
```

This pattern appears in 0/1 Knapsack, Complete Knapsack, Subset Sum, Partition Equal Subset Sum, Target Sum, Coin Change, Coin Change II, and many target-capacity problems.

The key idea is:

**Knapsack DP is about choosing items under constraints.**

**(PPT: Title Slide — DP Knapsack)**

## PPT 2 — Core Framework

When I think about knapsack DP, I usually break it down into five parts.

First, what the item is.
Second, what the capacity or target is.
Third, whether each item can be used once or multiple times.
Fourth, what the DP state means.
And finally, whether the goal is feasibility, counting, maximizing, or minimizing.

The most important question is:

```text
Can each item be used once, or unlimited times?
```

If each item can be used once, it is usually **0/1 knapsack**.

If each item can be used multiple times, it is usually **complete knapsack**.

The key idea is:

**Item usage controls the DP transition and loop direction.**

**(PPT: Core Framework — Item, Capacity, Usage, State)**

## PPT 3 — What Is 0/1 Knapsack?

In 0/1 knapsack, each item can be used at most once.

For every item, we make a decision:

```text
take it
or
skip it
```

For example:

```text
weights = [1, 3, 4]
values  = [15, 20, 30]
capacity = 4
```

If we take item 0 and item 1, the total weight is 4, and the total value is 35.

If we take item 2 only, the total weight is 4, but the value is 30.

So the best answer is 35.

The key idea is:

**0/1 means each item has only two choices: take or skip.**

**(PPT: 0/1 Knapsack Example)**

## PPT 4 — 0/1 Knapsack State and Recurrence

A classic 2D state is:

```text
dp[i][c] = maximum value using the first i items with capacity c
```

Here, item `i - 1` is the current item.

For each item, we have two choices.

If we skip it:

```text
dp[i - 1][c]
```

If we take it, and capacity allows:

```text
dp[i - 1][c - weight] + value
```

So the recurrence is:

```text
dp[i][c] = max(
    dp[i - 1][c],
    dp[i - 1][c - weight] + value
)
```

The key idea is:

**The recurrence comes from the take-or-skip decision.**

**(PPT: 0/1 Knapsack State + Recurrence)**

## PPT 5 — 0/1 Knapsack Code and Space Optimization

The 2D version is the easiest to understand.

```python
def knapsack_01(weights, values, capacity):
    n = len(weights)
    dp = [[0] * (capacity + 1) for _ in range(n + 1)]

    for i in range(1, n + 1):
        weight = weights[i - 1]
        value = values[i - 1]

        for c in range(capacity + 1):
            dp[i][c] = dp[i - 1][c]

            if c >= weight:
                dp[i][c] = max(
                    dp[i][c],
                    dp[i - 1][c - weight] + value
                )

    return dp[n][capacity]
```

But we can optimize it to 1D because each row only depends on the previous row.

For 0/1 knapsack, capacity must go backward:

```python
def knapsack_01_optimized(weights, values, capacity):
    dp = [0] * (capacity + 1)

    for i in range(len(weights)):
        weight = weights[i]
        value = values[i]

        for c in range(capacity, weight - 1, -1):
            dp[c] = max(dp[c], dp[c - weight] + value)

    return dp[capacity]
```

The key idea is:

**Backward iteration prevents using the same item more than once.**

**(PPT: 0/1 Knapsack 2D → 1D Optimization)**

## PPT 6 — Complete Knapsack

Complete knapsack is different.

In complete knapsack, each item can be used unlimited times.

This appears in problems like Coin Change and unbounded knapsack.

The recurrence looks similar:

```text
dp[c] = max(dp[c], dp[c - weight] + value)
```

But the loop direction changes.

For complete knapsack, capacity goes forward:

```python
def complete_knapsack(weights, values, capacity):
    dp = [0] * (capacity + 1)

    for i in range(len(weights)):
        weight = weights[i]
        value = values[i]

        for c in range(weight, capacity + 1):
            dp[c] = max(dp[c], dp[c - weight] + value)

    return dp[capacity]
```

Why forward?

Because `dp[c - weight]` is allowed to already include the current item.

The key idea is:

**Forward iteration allows unlimited reuse of the current item.**

**(PPT: Complete Knapsack — Unlimited Usage)**

## PPT 7 — Subset Sum and Partition Equal Subset Sum

Subset Sum is a 0/1 knapsack feasibility problem.

The state is:

```text
dp[s] = whether sum s can be formed
```

Code:

```python
def subset_sum(nums, target):
    dp = [False] * (target + 1)
    dp[0] = True

    for num in nums:
        for s in range(target, num - 1, -1):
            dp[s] = dp[s] or dp[s - num]

    return dp[target]
```

Because each number can be used once, we iterate backward.

Partition Equal Subset Sum reduces to Subset Sum.

If the total sum is odd, equal partition is impossible.

If the total sum is even, we only need to know whether some subset can form:

```text
total // 2
```

The key idea is:

**Many partition problems reduce to finding a subset with a target sum.**

**(PPT: Subset Sum + Partition Equal Subset Sum)**

## PPT 8 — Target Sum and Counting Knapsack

Knapsack is not only about max value.

Sometimes we count how many ways to form a target.

In **Target Sum**, we assign plus or minus signs to each number.

Let:

```text
P = sum of positive numbers
N = sum of negative numbers
```

We have:

```text
P - N = target
P + N = total
```

So:

```text
P = (target + total) / 2
```

Now the problem becomes:

```text
count subsets with sum P
```

Code:

```python
def find_target_sum_ways(nums, target):
    total = sum(nums)

    if abs(target) > total:
        return 0

    if (target + total) % 2 == 1:
        return 0

    subset_sum = (target + total) // 2
    dp = [0] * (subset_sum + 1)
    dp[0] = 1

    for num in nums:
        for s in range(subset_sum, num - 1, -1):
            dp[s] += dp[s - num]

    return dp[subset_sum]
```

The key idea is:

**Counting knapsack uses addition instead of max or boolean OR.**

**(PPT: Target Sum — Counting Subsets)**

## PPT 9 — Coin Change and Loop Order

Coin Change is a complete knapsack problem.

Each coin can be used unlimited times.

For minimum coins:

```python
def coin_change(coins, amount):
    INF = amount + 1
    dp = [INF] * (amount + 1)
    dp[0] = 0

    for coin in coins:
        for x in range(coin, amount + 1):
            dp[x] = min(dp[x], dp[x - coin] + 1)

    return dp[amount] if dp[amount] != INF else -1
```

For Coin Change II, we count combinations.

Order does not matter, so coins are in the outer loop:

```python
def change(amount, coins):
    dp = [0] * (amount + 1)
    dp[0] = 1

    for coin in coins:
        for x in range(coin, amount + 1):
            dp[x] += dp[x - coin]

    return dp[amount]
```

But for Combination Sum IV, order matters, so amount is in the outer loop.

The key idea is:

**Loop order decides whether we count combinations or ordered permutations.**

**(PPT: Coin Change + Loop Order)**

## PPT 10 — Senior-Level Explanation and Closing Insight

At senior level, I would explain knapsack DP like this.

Knapsack DP solves problems where we choose items under a capacity or target constraint.

The first question is what the item represents.

The second question is what the capacity represents.

The third question is whether each item can be used once or multiple times.

For 0/1 knapsack, each item can be used once, so capacity iterates backward.

For complete knapsack, each item can be reused, so capacity iterates forward.

The meaning of `dp` controls the transition.

If `dp` means feasibility, we use boolean OR.

If `dp` means count, we add the number of ways.

If `dp` means maximum value, we take max.

If `dp` means minimum cost, we take min.

The complexity is usually:

```text
O(number of items × capacity)
```

This is pseudo-polynomial because it depends on the numeric capacity.

The final insight is:

**Knapsack DP is not one template.
It is a modeling framework for choosing items under constraints.**

If you can explain the item, the capacity, the item usage rule, the meaning of `dp[c]`, the loop direction, and whether the goal is feasibility, counting, max value, or min cost, then you understand knapsack DP at an interview-ready level.

Thank you.

**(PPT: Closing Insight — Choose Items Under Constraints)**
