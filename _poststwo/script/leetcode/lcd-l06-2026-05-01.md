A stack is a data structure that follows **Last In, First Out** behavior.

That means the last item we push into the stack is the first item we pop out.

A typical stack flow looks like this:

```text
push item

→ pop when the current item resolves it
```

**Stack is useful when the most recent unresolved item should be processed first.**

## PPT 3 — Why Stack Works

Stack works because many problems have a nested or last-unresolved-first structure.

For example, in parentheses matching:

```text
{ [ ( ) ] }
```

The most recent opening bracket must be matched first.

So when we see a closing bracket, we only need to check the top of the stack.

Stack supports three basic operations:

```text
push → add item to the top
pop  → remove item from the top
peek → inspect the top item
```

This is why stack is useful for nested structures, expression parsing, undo behavior, DFS traversal, and monotonic stack problems.

The key idea is:

**The top of the stack represents the latest unresolved state.**

**(PPT: LIFO Behavior — Push / Pop / Peek)**

## PPT 4 — Valid Parentheses

A classic stack problem is **Valid Parentheses**.

The problem asks whether a bracket string is valid.

For example:

```text
"()"     → true
"()[]{}" → true
"(]"     → false
"([)]"   → false
```

The code is:

```python
def is_valid(s):
    stack = []
    pairs = {
        ")": "(",
        "]": "[",
        "}": "{"
    }

    for char in s:
        if char in pairs.values():
            stack.append(char)
        elif char in pairs:
            if not stack or stack[-1] != pairs[char]:
                return False

            stack.pop()

    return len(stack) == 0
```

The logic is simple.

Push opening brackets.
When a closing bracket appears, it must match the top of the stack.
At the end, the stack must be empty.

The key idea is:

**Closing brackets must match the most recent unmatched opening bracket.**

**(PPT: Valid Parentheses)**

## PPT 5 — Min Stack and Adjacent Duplicates

Stack is also useful in design problems.

For example, **Min Stack** asks us to support `push`, `pop`, `top`, and `getMin` all in **O(1)** time.

The idea is to use two stacks:

```text
normal stack → stores values
min stack    → stores minimum value at each depth
```

So `getMin` simply returns the top of the min stack.

Another simple stack problem is **Remove Adjacent Duplicates**.

For example:

```text
"abbaca" → "ca"
```

The code is:

```python
def remove_duplicates(s):
    stack = []

    for char in s:
        if stack and stack[-1] == char:
            stack.pop()
        else:
            stack.append(char)

    return "".join(stack)
```

Each character only needs to compare with the latest unresolved character.

The key idea is:

**Stack is powerful when the current item interacts with the most recent previous item.**

**(PPT: Min Stack + Remove Adjacent Duplicates)**

## PPT 6 — Expression Evaluation and Decode String

Stacks are also common in expression evaluation.

For Reverse Polish Notation:

```text
["2", "1", "+", "3", "*"]
```

This means:

```text
(2 + 1) * 3 = 9
```

The idea is:

Push operands.
When we see an operator, pop the latest two operands, calculate the result, and push it back.

Stack also works well for nested parsing problems like **Decode String**.

For example:

```text
"3[a]2[bc]" → "aaabcbc"
"3[a2[c]]" → "accaccacc"
```

When we enter a bracket, we save the current context.

When we close a bracket, we restore the most recent context and combine it with the decoded inner string.

The key idea is:

**Stack helps us save and restore nested context.**

**(PPT: Expression Evaluation + Decode String)**

## PPT 7 — Monotonic Stack

One of the most important stack patterns is **Monotonic Stack**.

A monotonic stack maintains elements in increasing or decreasing order.

It is used for:

```text
next greater element
next smaller element
previous greater element
previous smaller element
daily temperatures
stock span
largest rectangle in histogram
```

The general rule is:

When the current element breaks the monotonic order, we pop elements from the stack and resolve their answers.

For example, in a next greater element problem, we often maintain a decreasing stack of unresolved indexes.

When the current number is greater than the number at the top index, the current number becomes that index’s next greater element.

The key idea is:

**Monotonic stack stores unresolved candidates until a boundary appears.**

**(PPT: Monotonic Stack — Unresolved Candidates)**

## PPT 8 — Next Greater Element and Daily Temperatures

For **Next Greater Element**, the goal is to find the next greater value on the right.

```python
def next_greater_element(nums):
    result = [-1] * len(nums)
    stack = []

    for i, num in enumerate(nums):
        while stack and nums[stack[-1]] < num:
            index = stack.pop()
            result[index] = num

        stack.append(i)

    return result
```

The stack stores indexes whose answer has not been found yet.

**Daily Temperatures** is the same pattern, but instead of returning the value, we return the distance.

```text
answer[prev_index] = current_index - prev_index
```

So we store indexes, not just values.

The key idea is:

**Use indexes when the answer depends on position or distance.**

**(PPT: Next Greater Element + Daily Temperatures)**

## PPT 9 — Largest Rectangle in Histogram

A more advanced monotonic stack problem is **Largest Rectangle in Histogram**.

The goal is to find the largest rectangle area from bar heights.

The key insight is:

For each bar, find the first smaller bar on the left and the first smaller bar on the right.

Then:

```text
width = right_smaller_index - left_smaller_index - 1
area = height * width
```

The stack keeps bar indexes in increasing height order.

When the current height is smaller than the stack top, the popped bar has found its right boundary.

The new stack top becomes the left boundary.

A sentinel zero is often added at the end to force all remaining bars to be processed.

The key idea is:

**When a bar is popped, we finally know the maximum rectangle where that bar is the limiting height.**

**(PPT: Largest Rectangle in Histogram)**

## PPT 10 — Senior-Level Explanation and Closing Insight

At senior level, I would explain stack patterns like this.

A stack is a Last In, First Out data structure.

It is useful when the most recent unresolved item should be processed first.

For parentheses, the stack stores unmatched opening brackets.

For expression evaluation, it stores operands or previous context.

For Decode String, it stores nested parsing state.

For DFS, recursion uses the call stack implicitly, while iterative DFS uses an explicit stack.

For monotonic stack, the stack stores unresolved candidates and maintains an increasing or decreasing invariant.

When the current element breaks that invariant, we pop elements and resolve their answers.

Most stack algorithms are **O(n)** time and **O(n)** space.

For monotonic stack, even if there is a nested while loop, the total work is still **amortized O(n)** because each element is pushed once and popped once.

The final insight is:

**Stack Patterns are not just about append and pop.
They are about managing the most recent unresolved state.**

If you can explain what the stack stores, what invariant it maintains, when to push, when to pop, and what a popped element means, then you understand stack patterns at an interview-ready level.

Thank you.

**(PPT: Closing Insight — Most Recent Unresolved State)**
