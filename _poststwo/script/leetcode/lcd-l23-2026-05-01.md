## 23 Design Event Tracking System

## PPT 1 — Title Slide

Hello everyone.
Today I’d like to walk through a classic system design problem: **Design an Event Tracking System**.

An event tracking system is not just logging clicks.

At a deeper level, it is the data foundation for product analytics, funnels, experimentation, attribution, personalization, and business decisions.

The basic flow looks like this:

```text
User action
→ Client SDK creates event
→ Event collector validates it
→ Durable event log stores it
→ Stream and batch processors transform it
→ Analytics systems query it
```

The key idea is:

**Event Tracking System is a high-throughput analytics data pipeline.**

**(PPT: Title Slide — Design Event Tracking System)**

## PPT 2 — Core Requirements

For functional requirements, the system should track user and system events.

Common examples include page views, clicks, impressions, purchases, signups, searches, and backend events.

It should support web, mobile, and server-side tracking.

It should also support schema validation, real-time analytics, batch analytics, dashboards, funnels, attribution, experimentation, and data export.

For non-functional requirements, the system needs high write throughput, low ingestion latency, durable raw event storage, scalable processing, strong privacy controls, and cost-efficient long-term storage.

Most analytics can be eventually consistent.

But consent, deletion requests, experiment assignment, and critical conversion events need stronger correctness.

The key idea is:

**Analytics can be delayed, but privacy and critical events must be correct.**

**(PPT: Functional + Non-Functional Requirements)**

## PPT 3 — Event Schema

A good event schema is critical.

A typical event includes:

```text
eventId
eventName
userId
anonymousId
timestamp
receivedAt
properties
context
```

The `eventId` helps with deduplication.

The `eventName` tells us what happened.

The `userId` identifies a known user.

The `anonymousId` supports pre-login behavior tracking.

The `timestamp` is when the event happened on the client.

The `receivedAt` time is when the server received it.

The schema matters because bad event data can break analytics, dashboards, experiments, and downstream pipelines.

So I would use a schema registry to define allowed event names, required fields, property types, versions, and deprecation rules.

The key idea is:

**Event schema is the contract between product code and analytics systems.**

**(PPT: Event Schema + Schema Registry)**

---

## PPT 4 — Client SDK Design

The client SDK is important because events are generated at the edge.

The SDK should generate event IDs, attach user, session, device, and app context, buffer events locally, batch uploads, retry with backoff, and respect user consent.

For mobile clients, the app may go offline.

Battery usage and network usage matter.

Events may also arrive late.

For web clients, the page may close before events are sent.

So the SDK should avoid blocking page navigation and use browser-friendly mechanisms when possible.

Batching is also important.

Instead of sending every click immediately, the SDK can buffer events and send them in batches.

This reduces network overhead and improves reliability.

The key idea is:

**The SDK should collect events reliably without hurting user experience.**

**(PPT: Client SDK — Buffer, Batch, Retry, Consent)**

## PPT 5 — High-Level Architecture

At a high level, the architecture looks like this:

```text
Client SDK / Backend Service
→ Event Collector
→ Validation and Enrichment
→ Message Queue / Event Log
→ Stream Processing
→ Raw Event Storage
→ Analytics Warehouse
→ Dashboard / Experiment / Attribution
```

The event collector receives events, authenticates clients, rate limits traffic, validates payloads, and adds server-side metadata.

Then it writes events to a durable event log, such as Kafka, Pulsar, or Kinesis.

Once the event is durably written, the collector can return success.

Heavy processing should happen asynchronously.

The stream processor consumes events, deduplicates them, enriches them, computes real-time metrics, and writes to downstream stores.

The key idea is:

**Collectors should ingest quickly; processors should do heavier work asynchronously.**

**(PPT: High-Level Architecture Diagram)**

## PPT 6 — Ingestion, Deduplication, and Ordering

The ingestion flow should be lightweight.

```text
Receive event
→ Authenticate source
→ Validate schema
→ Add received timestamp
→ Enrich context
→ Write to durable event log
→ Return success
```

The system should assume duplicates will happen.

Duplicates can come from client retries, network timeouts, SDK resend, collector retry, or stream processor retry.

Every event should include a unique event ID.

Deduplication can use a short-term cache, a persistent dedupe table for critical events, and idempotent downstream writes.

Ordering is also not guaranteed.

For example, a purchase event may arrive before an add-to-cart event.

So we should store both event time and received time.

For analytics, stream processors can use watermarks and allowed lateness windows.

The key idea is:

**Event systems should expect duplicates, retries, and late arrivals by design.**

**(PPT: Deduplication + Late Events)**

---

## PPT 7 — Stream Processing and Batch Processing

The system usually needs both stream processing and batch processing.

Stream processing supports near-real-time use cases.

Examples include live dashboards, real-time alerts, experiment metrics, fraud detection, personalization, and operational monitoring.

A typical stream flow is:

```text
Kafka topic
→ Stream processor
→ Deduplicate and aggregate by time window
→ Write to real-time analytics store
```

Batch processing supports long-term and high-confidence analytics.

Examples include daily reporting, attribution, retention, sessionization, model training, data quality checks, and backfills.

Raw events should be stored durably in object storage.

This allows replay, backfill, data correction, auditing, and rebuilding downstream tables.

The key idea is:

**Hot path gives freshness. Cold path gives reliability and replayability.**

**(PPT: Stream Processing + Batch Processing)**

---

## PPT 8 — Storage Design and Analytics Use Cases

I would use multiple storage systems.

Raw events go to object storage, such as S3.

They should be partitioned by date, hour, event name, or source.

Real-time aggregates can go to systems like ClickHouse, Druid, Pinot, or Elasticsearch.

These are good for fast dashboards and time-range queries.

Curated historical tables go to a data warehouse, such as Snowflake, BigQuery, Redshift, or Hive.

These support ad hoc SQL, historical reporting, and model training.

This system powers many analytics use cases.

Funnels track steps like:

```text
view_product
→ add_to_cart
→ checkout_started
→ purchase_completed
```

It can also support retention, attribution, experimentation, and product analytics.

The key idea is:

**Raw storage preserves truth; processed tables make analytics fast and usable.**

**(PPT: Storage Layers + Analytics Use Cases)**

## PPT 9 — Privacy, Data Quality, and Failure Handling

Event tracking can easily collect sensitive data.

Events may include user IDs, IP addresses, device IDs, location, emails, payment metadata, or financial signals.

So the system must enforce consent management, data minimization, PII redaction, encryption, access control, audit logs, retention policies, and user deletion support.

A very important rule is:

```text
Do not allow arbitrary PII in event properties.
```

Data quality is also critical.

Common issues include missing required fields, wrong property types, duplicate events, late events, inconsistent event names, client clock skew, bot traffic, and version mismatch.

To handle this, I would use schema validation, event versioning, dead-letter queues, data quality dashboards, bot filtering, sampling audits, and anomaly alerts.

The key idea is:

**An event tracking system is only useful if the data is trustworthy and privacy-safe.**

**(PPT: Privacy + Data Quality + Failure Handling)**

## PPT 10 — Senior-Level Explanation and Closing Insight

At senior level, I would explain an event tracking system like this.

Event tracking is a high-throughput analytics pipeline.

Events are generated by web clients, mobile clients, and backend services.

SDKs generate event IDs, attach context, buffer events, batch uploads, retry failures, and respect consent.

Collectors validate events, enrich them, and write them to a durable event log.

Raw events are stored in object storage so the system can replay, backfill, audit, and rebuild downstream tables.

Stream processing powers near-real-time dashboards, alerts, experiments, and personalization.

Batch processing powers reliable historical analytics, sessionization, attribution, retention, and model training.

Data quality requires schema registry, event versioning, dead-letter queues, deduplication, and monitoring.

Privacy requires consent enforcement, data minimization, PII redaction, encryption, access control, retention, and deletion support.

The final insight is:

**Event Tracking System is not just logging user actions.
It is the data infrastructure behind analytics, experimentation, attribution, personalization, and product decisions.**

The goal is to provide reliable, trusted, scalable behavioral data while balancing accuracy, latency, privacy, and cost.

Thank you.

**(PPT: Closing Insight — Reliable Behavioral Data Infrastructure)**
