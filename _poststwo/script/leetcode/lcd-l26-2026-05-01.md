## PPT 1 — Title Slide

Hello everyone.
Today I’d like to walk through a classic system design problem: **Design a Web Crawler**.

A web crawler is not just a program that downloads web pages.

At a deeper level, it is a distributed system for discovering, scheduling, fetching, parsing, deduplicating, storing, and indexing web content.

The basic flow looks like this:

```text
Seed URLs
→ URL frontier
→ Scheduler
→ Crawl workers
→ HTML parser
→ URL extractor
→ Deduplication
→ Storage and index
```

The key idea is:

**A web crawler is a distributed URL scheduling and fetching pipeline.**

**(PPT: Title Slide — Design Web Crawler)**

## PPT 2 — Core Requirements

For functional requirements, the crawler should continuously crawl web pages, discover new URLs, extract links and metadata, avoid duplicate crawling, respect robots.txt, support recrawling, store crawled content, and support large-scale distributed crawling.

For non-functional requirements, the system needs high throughput, fault tolerance, scalable workers, low duplicate rate, efficient storage usage, fairness toward websites, and the ability to resume after failures.

Eventual consistency is acceptable.

The web is constantly changing, so we do not need every worker to have a perfectly synchronized view.

But we do need strong coordination for crawl scheduling, deduplication, and politeness.

The key idea is:

**The crawler must scale aggressively while behaving responsibly toward websites.**

**(PPT: Functional + Non-Functional Requirements)**

## PPT 3 — High-Level Architecture

At a high level, I would separate the crawler into independent components.

```text
Seed URLs
→ URL Frontier
→ Scheduler
→ Crawl Workers
→ Parser and Extractor
→ Deduplication
→ Storage and Index
```

The **URL Frontier** stores pending URLs and controls crawl priority.

The **Scheduler** assigns URLs to workers and enforces politeness rules.

The **Crawl Workers** fetch web pages, handle redirects, retries, timeouts, and HTTP responses.

The **Parser** extracts links, titles, metadata, and text.

The **Deduplication layer** prevents duplicate work.

The **Storage and Index layer** stores raw HTML, metadata, link graph, and searchable content.

The key idea is:

**Separating scheduling, fetching, parsing, deduplication, and storage makes the crawler scalable and fault-tolerant.**

**(PPT: High-Level Architecture Diagram)**

## PPT 4 — URL Discovery and Normalization

A crawler starts with seed URLs.

But most URLs are discovered recursively from fetched pages.

The URL discovery flow looks like this:

```text
Fetch HTML
→ Parse DOM
→ Extract hyperlinks
→ Normalize URLs
→ Deduplicate
→ Push into frontier
```

URL normalization is very important.

For example, we may normalize HTTP to HTTPS, remove fragments, canonicalize query parameters, lowercase the hostname, and remove duplicate slashes.

Without normalization, the same page may appear as many different URLs.

That can cause duplicate crawling and huge storage waste.

The key idea is:

**New URLs must be normalized and deduplicated before entering the frontier.**

**(PPT: URL Discovery + Normalization)**

## PPT 5 — URL Frontier and Scheduling

The URL frontier is the heart of the crawler.

It controls crawl order, priority, freshness, fairness, and throughput.

A simple crawler may use FIFO scheduling, similar to breadth-first search.

But a production crawler usually needs priority-based scheduling.

Higher priority may go to popular pages, frequently updated pages, important domains, or pages with high authority.

Host-based scheduling is also important.

We should avoid sending too many requests to the same website at once.

For example:

```text
example.com → crawl every 5 seconds
news.com → crawl every 500 milliseconds
```

The key idea is:

**The frontier is not just a queue. It controls freshness, fairness, and crawl priority.**

**(PPT: URL Frontier + Scheduling Strategy)**

## PPT 6 — Fetching Pages and Politeness

A crawl worker usually follows this flow:

```text
Get URL from frontier
→ Check robots.txt
→ DNS lookup
→ HTTP fetch
→ Handle redirects
→ Store response
→ Send content to parser
```

Fetching has many practical concerns.

We need to handle redirects, timeouts, compression, retries, TLS, HTTP status codes, and user-agent management.

Retries should use exponential backoff.

We should retry transient failures, but avoid retrying forever.

Politeness is also critical.

A production crawler must respect robots.txt, enforce per-domain rate limits, limit concurrent requests, randomize intervals when needed, and identify itself with a proper user agent.

The key idea is:

**Crawl workers should fetch efficiently, but the scheduler must keep crawling polite.**

**(PPT: Fetching Flow + robots.txt + Rate Limits)**

## PPT 7 — Deduplication

Deduplication is critical at internet scale.

Duplicates happen because of multiple links to the same page, URL aliases, tracking parameters, redirect chains, session IDs, and mirrored content.

There are two major deduplication layers.

The first is **URL-level deduplication**.

This prevents recrawling the same normalized URL.

The second is **content-level deduplication**.

This detects identical or near-identical pages after fetching.

For exact duplicate content, we can use:

```text
hash(page_content)
```

For near duplicates, we can use techniques like SimHash or MinHash.

A Bloom filter can help quickly check whether a URL has likely been seen before.

The key idea is:

**Deduplication reduces wasted fetches, wasted storage, and repeated indexing.**

**(PPT: URL Dedup + Content Dedup)**

## PPT 8 — Parsing, Storage, and Indexing

After a page is fetched, the parser extracts structured information.

This may include links, title, metadata, structured data, images, body text, keywords, and anchor text.

Parsing can be difficult because HTML may be malformed.

Some pages are JavaScript-rendered.

Some websites generate infinite calendars or dynamic URLs.

For JavaScript-heavy pages, we can use a headless browser or rendering service, but this is expensive.

So I would only use full rendering selectively.

For storage, I would separate different data types.

Raw HTML goes to object storage.

Crawl metadata goes to a database.

Searchable content goes to a search index.

The link graph can be stored separately for ranking and analysis.

The key idea is:

**Raw pages, crawl metadata, search index, and link graph serve different purposes and should be stored separately.**

**(PPT: Parsing + Storage + Search Indexing)**

## PPT 9 — Distributed Scaling and Recrawling

The web is huge, so the crawler must be distributed.

I would use distributed URL queues, stateless crawl workers, asynchronous fetching, and host-based partitioning.

Partitioning by host helps enforce politeness and improves locality.

For example, one worker group may handle one set of domains, while another group handles a different set.

The crawler also needs recrawling.

Web crawling is not one-time.

Pages change over time.

A news homepage may need to be crawled every minute.

A blog archive may only need to be crawled once a week.

Recrawl frequency can depend on popularity, historical update frequency, HTTP cache headers, and page importance.

The key idea is:

**A production crawler continuously balances coverage, freshness, and cost.**

**(PPT: Distributed Crawling + Freshness + Recrawl)**

## PPT 10 — Senior-Level Explanation and Closing Insight

At senior level, I would explain a web crawler like this.

A web crawler is a distributed pipeline for URL discovery, scheduling, fetching, parsing, deduplication, storage, and indexing.

Seed URLs enter a URL frontier.

The scheduler chooses which URLs to crawl next, while enforcing priority, fairness, and politeness.

Crawl workers fetch pages asynchronously, respect robots.txt, handle retries and redirects, and send responses to parsers.

Parsers extract links and metadata, normalize new URLs, and push new candidates back into the frontier.

Deduplication is critical, using normalized URLs, Bloom filters, content hashing, and near-duplicate detection.

Raw HTML, crawl metadata, search indexes, and link graphs should be stored separately.

The system must handle failures through persistent queues, retries, checkpointing, dead-letter queues, and idempotent processing.

The final insight is:

**A web crawler is not just downloading pages.
It is a large-scale distributed system built around URL frontier, scheduling, politeness, deduplication, parsing, and indexing.**

The goal is to discover, fetch, and index web content efficiently and responsibly at internet scale.

Thank you.

**(PPT: Closing Insight — Distributed URL Scheduling and Fetching System)**
