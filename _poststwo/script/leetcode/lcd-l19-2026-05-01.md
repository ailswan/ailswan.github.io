Hello everyone.
Today I’d like to walk through another important data structure pattern: **Trie**.

A Trie is also called a **prefix tree**.

It is designed to store strings so that prefix queries become efficient.

The basic idea is:

```text
Each node represents a character
→ Paths represent prefixes
→ A complete path represents a word
→ Shared prefixes are stored only once
```

Trie is commonly used for prefix search, autocomplete, word dictionary, spell checking, word search, and string matching problems.

The key idea is:

**Trie organizes words by shared prefixes.**

## PPT 3 — Why Trie Works

Suppose we store these words:

```text
car
cat
care
dog
```

The words `car`, `cat`, and `care` share the prefix `"ca"`.

Instead of storing that prefix repeatedly, Trie stores it once.

The structure looks like:

```text
root
 ├── c
 │   └── a
 │       ├── r
 │       │   └── e
 │       └── t
 └── d
     └── o
         └── g
```

Each path from root to a node represents a prefix.

Some nodes mark the end of a complete word.

For example, the node for `"r"` can mark `"car"`, while the node for `"e"` marks `"care"`.

The key idea is:

**Trie saves shared prefixes and marks where full words end.**

**(PPT: Trie Structure Example)**

## PPT 4 — Trie Node Design

A Trie node usually needs two things.

First, a map from character to child node.

Second, a boolean flag to indicate whether this node ends a word.

In Python, a node can look like this:

```python
class TrieNode:
    def __init__(self):
        self.children = {}
        self.is_word = False
```

The `children` dictionary stores outgoing edges.

For example:

```text
children["a"] = next TrieNode
```

The `is_word` flag is important.

Without it, we cannot distinguish a prefix from a complete word.

For example, if `"apple"` exists, `"app"` may be a prefix, but not necessarily a word.

The key idea is:

**A Trie node stores next characters and whether a word ends here.**

**(PPT: Trie Node = Children + is_word)**

## PPT 5 — Insert Operation

To insert a word, we start from the root.

For each character, we check whether the child exists.

If not, we create a new Trie node.

At the end, we mark the final node as a word.

```python
class Trie:
    def __init__(self):
        self.root = TrieNode()

    def insert(self, word):
        node = self.root

        for ch in word:
            if ch not in node.children:
                node.children[ch] = TrieNode()

            node = node.children[ch]

        node.is_word = True
```

For a word of length `L`, insertion takes **O(L)** time.

It does not depend on how many words already exist, except through shared nodes.

The key idea is:

**Insert walks character by character and creates missing nodes.**

**(PPT: Insert Word into Trie)**

## PPT 6 — Search Operation

To search for a complete word, we again start from the root.

For each character, we move to the corresponding child.

If a character is missing, the word does not exist.

At the end, we must check `is_word`.

```python
def search(self, word):
    node = self.root

    for ch in word:
        if ch not in node.children:
            return False

        node = node.children[ch]

    return node.is_word
```

This final check is important.

If we inserted `"apple"`, then searching for `"app"` should return false unless `"app"` was also inserted.

The key idea is:

**Search needs both path existence and word-ending confirmation.**

**(PPT: Search Complete Word)**

## PPT 7 — Prefix Search

Prefix search is similar to word search, but we do not require `is_word` at the end.

This is usually called `startsWith`.

```python
def starts_with(self, prefix):
    node = self.root

    for ch in prefix:
        if ch not in node.children:
            return False

        node = node.children[ch]

    return True
```

If the prefix path exists, then at least one word starts with that prefix.

This is why Trie is useful for autocomplete.

Once we find the prefix node, we can run DFS from that node to collect all words under it.

The key idea is:

**Prefix search only checks whether the prefix path exists.**

**(PPT: startsWith Prefix Query)**

## PPT 8 — Word Dictionary With Wildcard

A common advanced Trie problem is **Design Add and Search Words Data Structure**.

Here, the search word may contain `"."`, which can match any character.

For example:

```text
bad
dad
mad

search(".ad") → true
search("b..") → true
```

This requires DFS.

When the current character is normal, we follow that child.

When the current character is `"."`, we try all children.

```python
def search_word(node, word, index):
    if index == len(word):
        return node.is_word

    ch = word[index]

    if ch == ".":
        for child in node.children.values():
            if search_word(child, word, index + 1):
                return True
        return False

    if ch not in node.children:
        return False

    return search_word(node.children[ch], word, index + 1)
```

The key idea is:

**Wildcard search turns Trie lookup into DFS over possible branches.**

**(PPT: Wildcard Trie Search)**

## PPT 9 — Word Search II

Trie is also very useful for **Word Search II**.

The problem gives a board of characters and a list of words.

We need to find which words exist in the board.

A brute-force approach searches every word separately.

That can repeat a lot of work.

Instead, we build a Trie from all words.

Then we run DFS on the board.

As we move through cells, we follow Trie edges.

If the current path is not in the Trie, we stop early.

This is powerful pruning.

When we reach a Trie node marked as a word, we found one answer.

The key idea is:

**Trie lets board DFS stop early when no word has the current prefix.**

**(PPT: Word Search II — Trie + DFS Pruning)**

## PPT 10 — Senior-Level Explanation and Closing Insight

At senior level, I would explain Trie like this.

Trie is a tree-based data structure for storing strings by prefix.

Each node represents a character position.

Each edge represents a character transition.

The root represents the empty prefix.

A path from root to a node represents a prefix.

A boolean flag marks whether that prefix is also a complete word.

Insert, search, and prefix search usually take **O(L)** time, where `L` is the length of the word or prefix.

Trie is useful when we have many string queries, especially prefix queries.

It is common in autocomplete, dictionary lookup, wildcard search, Word Search II, and prefix-based matching.

The main trade-off is memory.

Trie can use more memory than a hash set because each node stores children pointers.

But it gives efficient prefix traversal and pruning.

The final insight is:

**Trie is not just a tree of characters.
It is a data structure for sharing prefixes and accelerating prefix-based search.**

If you can explain what each node stores, how insert works, why search checks `is_word`, how prefix search differs from full word search, and why Trie helps prune DFS, then you understand Trie at an interview-ready level.
