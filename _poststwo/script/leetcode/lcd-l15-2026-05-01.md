## PPT 1 — Title Slide

Hello everyone.
Today I’d like to walk through another important dynamic programming pattern: **Grid DP**.

Grid DP is used when the input is a two-dimensional matrix, and the answer for each cell depends on nearby cells.

The basic idea is:

```text
Define dp[r][c]
→ Understand movement direction
→ Compute from neighboring cells
→ Fill the table in dependency order
```

Common examples include Unique Paths, Minimum Path Sum, Triangle, Maximal Square, Dungeon Game, Falling Path Sum, and obstacle-based path problems.

The key idea is:

**Grid DP treats each cell as a subproblem.**

When I think about Grid DP, I usually break it down into five parts.

First, what `dp[r][c]` means.
Second, what movement directions are allowed.
Third, which neighboring cells the current cell depends on.
Fourth, how to handle boundaries and obstacles.
And finally, whether space can be optimized.

```text
What does dp[r][c] represent?
```

Depending on the problem, it may mean:

```text
number of ways to reach this cell
minimum cost to reach this cell
maximum score ending at this cell
largest square ending at this cell
minimum health needed before entering this cell
```

The key idea is:

**The state meaning decides the recurrence.**

## PPT 3 — Basic Grid DP Model

In many basic grid DP problems, movement is limited to:

```text
right
down
```

If we look backward, that means each cell can only be reached from:

```text
top
left
```

So a common recurrence is:

```text
dp[r][c] = combine(dp[r - 1][c], dp[r][c - 1])
```

If the problem counts paths, combine means addition.

If the problem finds minimum cost, combine means taking the minimum.

For example:

```text
Path counting:
dp[r][c] = dp[r - 1][c] + dp[r][c - 1]

Minimum cost:
dp[r][c] = grid[r][c] + min(dp[r - 1][c], dp[r][c - 1])
```

The key idea is:

**Movement direction defines dependency direction.**

**(PPT: Cell Depends on Top and Left)**

## PPT 4 — Why Iteration Order Matters

Iteration order is critical in grid DP.

If `dp[r][c]` depends on the top and left cells, then we should compute the table from:

```text
top-left to bottom-right
```

So by the time we reach cell `(r, c)`, both dependencies are already computed.

```python
for r in range(rows):
    for c in range(cols):
        compute dp[r][c]
```

But some problems depend on future cells.

For example, Dungeon Game depends on the right and down cells.

In that case, we compute from:

```text
bottom-right to top-left
```

The key idea is:

**A DP table must be filled after its dependencies are available.**

**(PPT: Forward DP vs Reverse DP)**

## PPT 5 — Unique Paths

A classic grid DP problem is **Unique Paths**.

A robot starts at the top-left corner and wants to reach the bottom-right corner.

It can only move right or down.

State definition:

```text
dp[r][c] = number of ways to reach cell (r, c)
```

Recurrence:

```text
dp[r][c] = dp[r - 1][c] + dp[r][c - 1]
```

Code:

```python
def unique_paths(m, n):
    dp = [[0] * n for _ in range(m)]

    for r in range(m):
        dp[r][0] = 1

    for c in range(n):
        dp[0][c] = 1

    for r in range(1, m):
        for c in range(1, n):
            dp[r][c] = dp[r - 1][c] + dp[r][c - 1]

    return dp[m - 1][n - 1]
```

The first row and first column each have exactly one path.

The key idea is:

**Path counting adds ways from valid previous cells.**

**(PPT: Unique Paths — Counting Ways)**

## PPT 6 — Obstacles and Space Optimization

Unique Paths can also include obstacles.

If a cell is blocked, it contributes zero paths.

```text
if obstacle:
    dp[r][c] = 0
```

Otherwise, it still comes from top and left.

Grid DP can often be optimized to one row.

For Unique Paths:

```python
def unique_paths(m, n):
    dp = [1] * n

    for r in range(1, m):
        for c in range(1, n):
            dp[c] += dp[c - 1]

    return dp[-1]
```

Here, before update:

```text
dp[c] means top
dp[c - 1] means left
```

The key idea is:

**In 1D grid DP, each cell may represent both previous row and current row depending on update timing.**

**(PPT: Obstacles + 1D DP Optimization)**

## PPT 7 — Minimum Path Sum

Another classic problem is **Minimum Path Sum**.

We are given a grid of non-negative numbers and need the minimum path sum from top-left to bottom-right.

State definition:

```text
dp[r][c] = minimum path sum to reach cell (r, c)
```

Recurrence:

```text
dp[r][c] = grid[r][c] + min(dp[r - 1][c], dp[r][c - 1])
```

Code:

```python
def min_path_sum(grid):
    rows = len(grid)
    cols = len(grid[0])

    dp = [[0] * cols for _ in range(rows)]
    dp[0][0] = grid[0][0]

    for r in range(1, rows):
        dp[r][0] = dp[r - 1][0] + grid[r][0]

    for c in range(1, cols):
        dp[0][c] = dp[0][c - 1] + grid[0][c]

    for r in range(1, rows):
        for c in range(1, cols):
            dp[r][c] = grid[r][c] + min(
                dp[r - 1][c],
                dp[r][c - 1]
            )

    return dp[rows - 1][cols - 1]
```

The key idea is:

**Minimum path problems take the best previous path and add the current cell cost.**

**(PPT: Minimum Path Sum)**

## PPT 8 — Triangle and Maximal Square

Grid DP is not always a rectangle.

For **Triangle**, each cell depends on two adjacent cells below it.

A clean solution is bottom-up:

```python
def minimum_total(triangle):
    dp = triangle[-1][:]

    for r in range(len(triangle) - 2, -1, -1):
        for c in range(len(triangle[r])):
            dp[c] = triangle[r][c] + min(dp[c], dp[c + 1])

    return dp[0]
```

For **Maximal Square**, the state is different:

```text
dp[r][c] = side length of the largest all-one square
ending at cell (r, c)
```

If the current cell is 1:

```text
dp[r][c] = 1 + min(top, left, top-left)
```

The key idea is:

**Different grid problems may use different neighbors and different state meanings.**

**(PPT: Triangle + Maximal Square)**

## PPT 9 — Dungeon Game and Reverse Grid DP

Some grid DP problems must be solved backward.

A classic example is **Dungeon Game**.

The knight starts at the top-left cell, but we need the minimum initial health required to survive until the bottom-right cell.

Forward DP is hard because current path sum is not enough.

We need to know future survival requirements.

So we define:

```text
dp[r][c] = minimum health needed before entering cell (r, c)
```

The recurrence is:

```text
need = min(dp[r + 1][c], dp[r][c + 1]) - dungeon[r][c]
dp[r][c] = max(1, need)
```

We compute from bottom-right to top-left.

The key idea is:

**When the current decision depends on future survival, reverse DP may be more natural.**

**(PPT: Dungeon Game — Reverse DP)**

## PPT 10 — Senior-Level Explanation and Closing Insight

At senior level, I would explain Grid DP like this.

Grid DP applies when the input is a matrix and each cell can be treated as a subproblem.

The first step is defining what `dp[r][c]` means.

It may represent path count, minimum cost, maximum score, square size, or required health.

Then I identify movement directions.

If movement is right and down, the current cell usually depends on top and left.

If the problem depends on future cells, I may need reverse iteration.

Boundary handling is important because the first row and first column often have only one predecessor.

Obstacles are usually invalid states.

For path counting, blocked cells contribute zero ways.

For minimum cost, blocked cells may be treated as infinity.

The usual complexity is:

```text
Time: O(mn)
Space: O(mn)
```

With row compression, space can become **O(n)**.

If input mutation is allowed, some problems can be solved in-place with **O(1)** extra space.

The final insight is:

**Grid DP is not just filling a 2D table.
It is defining cell dependencies from movement rules.**

If you can explain what each cell state means, which neighbors it depends on, why the iteration order is correct, how boundaries and obstacles are handled, and whether space optimization is safe, then you understand Grid DP at an interview-ready level.

Thank you.

**(PPT: Closing Insight — Cell State + Dependency Direction)**
