Today I’d like to walk through an important graph pattern: **Graph Cycle Detection**.

Cycle detection asks one simple question:

```text
Does this graph contain a cycle?
```

A cycle means we can start from a node, follow edges, and eventually return to a previously visited node.

This pattern appears in Course Schedule, dependency validation, build systems, package managers, deadlock detection, redundant connection, grid cycle detection, and graph validity problems.

The key idea is:

**Cycle detection depends heavily on whether the graph is directed or undirected.**

The key idea is:

**The graph type decides the cycle detection strategy.**

## PPT 3 — Cycle in Undirected Graph

In an undirected graph, a cycle exists if DFS reaches a visited neighbor that is not the parent.

For example:

```text
0 -- 1
|    |
3 -- 2
```

This graph has a cycle:

```text
0 → 1 → 2 → 3 → 0
```

The key rule is:

```text
If neighbor is visited
and neighbor is not parent,
then cycle exists.
```

Why do we need the parent check?

Because every undirected edge can be seen from both sides.

If we go from `0` to `1`, then from `1` we can see `0` again.

That does not mean cycle.

That is just the edge back to the parent.

The key idea is:

**In undirected graphs, visited neighbor is only a cycle if it is not the parent.**

**(PPT: Undirected Cycle — Parent Tracking)**

## PPT 4 — Undirected DFS Template

The standard undirected DFS template passes the parent node.

```python
from collections import defaultdict

def has_cycle_undirected(n, edges):
    graph = defaultdict(list)

    for u, v in edges:
        graph[u].append(v)
        graph[v].append(u)

    visited = set()

    def dfs(node, parent):
        visited.add(node)

        for neighbor in graph[node]:
            if neighbor not in visited:
                if dfs(neighbor, node):
                    return True
            elif neighbor != parent:
                return True

        return False

    for node in range(n):
        if node not in visited:
            if dfs(node, -1):
                return True

    return False
```

We loop through all nodes because the graph may be disconnected.

If we only start from node zero, we may miss a cycle in another component.

The key idea is:

**For disconnected graphs, cycle detection must check every component.**

**(PPT: Undirected DFS Code)**

## PPT 5 — Cycle in Directed Graph

Directed graph cycle detection is different.

In a directed graph, parent tracking is not enough.

We need to know whether a node is currently in the recursion path.

A common approach uses three states:

```text
0 = unvisited
1 = visiting
2 = visited
```

If DFS reaches a node with state `1`, we found a back edge.

That means there is a directed cycle.

```text
0 → 1 → 2
    ↑   |
    |___|
```

Here, DFS can go from `1` to `2`, then back to `1`.

Since `1` is still in the recursion path, this is a cycle.

The key idea is:

**In directed graphs, a cycle means reaching a node currently being visited.**

**(PPT: Directed Cycle — Three States)**

## PPT 6 — Directed DFS Template

Here is the standard directed graph DFS template.

```python
from collections import defaultdict

def has_cycle_directed(n, edges):
    graph = defaultdict(list)

    for u, v in edges:
        graph[u].append(v)

    state = [0] * n

    def dfs(node):
        if state[node] == 1:
            return True

        if state[node] == 2:
            return False

        state[node] = 1

        for neighbor in graph[node]:
            if dfs(neighbor):
                return True

        state[node] = 2
        return False

    for node in range(n):
        if state[node] == 0:
            if dfs(node):
                return True

    return False
```

When entering a node, we mark it as visiting.

After all neighbors are processed, we mark it as fully visited.

The key idea is:

**Visiting means currently on the recursion stack. Visited means fully processed.**

**(PPT: Directed DFS State Template)**

## PPT 7 — Course Schedule

Course Schedule is the classic directed cycle detection problem.

Each prerequisite:

```text
[course, prerequisite]
```

means:

```text
To take course, we must first take prerequisite.
```

So the edge direction is:

```text
prerequisite → course
```

If the graph has a cycle, the courses depend on each other circularly.

That means it is impossible to finish all courses.

```python
from collections import defaultdict

def can_finish(num_courses, prerequisites):
    graph = defaultdict(list)

    for course, prereq in prerequisites:
        graph[prereq].append(course)

    state = [0] * num_courses

    def dfs(course):
        if state[course] == 1:
            return False
        if state[course] == 2:
            return True

        state[course] = 1

        for next_course in graph[course]:
            if not dfs(next_course):
                return False

        state[course] = 2
        return True

    for course in range(num_courses):
        if state[course] == 0:
            if not dfs(course):
                return False

    return True
```

The key idea is:

**Course Schedule is dependency cycle detection.**

**(PPT: Course Schedule — Circular Dependency)**

## PPT 8 — Topological Sort Cycle Detection

Topological sort can also detect cycles in a directed graph.

The idea is:

```text
Compute indegree
Start with nodes that have indegree zero
Process nodes
Reduce neighbors' indegrees
```

If the graph has no cycle, we should be able to process all nodes.

If some nodes remain unprocessed, they are part of or blocked by a cycle.

```python
from collections import defaultdict, deque

def has_cycle_topological(n, edges):
    graph = defaultdict(list)
    indegree = [0] * n

    for u, v in edges:
        graph[u].append(v)
        indegree[v] += 1

    queue = deque([i for i in range(n) if indegree[i] == 0])
    processed = 0

    while queue:
        node = queue.popleft()
        processed += 1

        for neighbor in graph[node]:
            indegree[neighbor] -= 1

            if indegree[neighbor] == 0:
                queue.append(neighbor)

    return processed != n
```

The key idea is:

**If topological sort cannot process all nodes, the directed graph has a cycle.**

**(PPT: Topological Sort Cycle Detection)**

## PPT 9 — Union Find and Grid Cycle Detection

For undirected graphs, Union Find is another strong approach.

For each edge `(u, v)`:

```text
if find(u) == find(v):
    cycle exists
else:
    union(u, v)
```

This works because if two nodes are already connected, adding another edge between them creates a cycle.

This pattern is useful for **Redundant Connection**.

Grid cycle detection can also be modeled as undirected graph cycle detection.

Each cell is a node.

Adjacent cells with the same character are connected.

During DFS, if we reach a visited cell that is not the parent cell, then a cycle exists.

The key idea is:

**Union Find detects undirected cycles by checking whether an edge connects nodes already in the same component.**

**(PPT: Union Find + Grid Cycle Detection)**

## PPT 10 — Senior-Level Explanation and Closing Insight

At senior level, I would explain graph cycle detection like this.

Graph cycle detection determines whether a graph contains a path that returns to a previously visited node.

The first distinction is whether the graph is undirected or directed.

For undirected graphs, DFS uses parent tracking.

If DFS reaches a visited neighbor that is not the parent, then there is a cycle.

Union Find is another good option for undirected graphs, especially when edges are processed one by one.

If an edge connects two nodes already in the same component, adding that edge creates a cycle.

For directed graphs, parent tracking is not enough.

We need recursion state.

DFS uses three states: unvisited, visiting, and visited.

If DFS reaches a visiting node, that is a back edge, so a directed cycle exists.

Topological sort can also detect directed cycles.

If we cannot process all nodes by repeatedly removing indegree-zero nodes, then the graph has a cycle.

The final insight is:

**Graph Cycle Detection is not just “visited means cycle.”
It is about using the right meaning of visited for the graph type.**
