## PPT 3 — Why Backtracking Works

Backtracking works because many problems can be modeled as a decision tree.

For example, when generating subsets from `[1, 2, 3]`, at each step we decide whether to include or skip an element.

Each path in the decision tree represents one possible result.

The core pattern is:

```text
choose
explore
unchoose
```

In code, that usually looks like:

```python
path.append(choice)
backtrack(...)
path.pop()
```

The undo step is very important.

Because we usually reuse the same `path` object across recursive calls.

After exploring one branch, we must restore the previous state before trying the next branch.

The key idea is:

**Backtracking explores one branch, restores state, then explores another branch.**

**(PPT: Decision Tree + Choose / Explore / Unchoose)**

## PPT 4 — Basic Backtracking Template

A standard backtracking template looks like this:

```python
def backtrack(path, choices):
    if is_solution(path):
        result.append(path[:])
        return

    for choice in choices:
        if not is_valid(choice):
            continue

        path.append(choice)
        backtrack(path, next_choices)
        path.pop()
```

Most backtracking problems need seven components.

A current path.
A result list.
Available choices.
A base case.
A validity check.
A recursive call.
And an undo step.

One common mistake is appending `path` directly to the result.

Because `path` is mutable, we should append a copy:

```python
result.append(path[:])
```

The key idea is:

**The path represents the current partial solution.**

## PPT 5 — Subsets and Combinations

For **Subsets**, every partial path is already a valid answer.

```python
def subsets(nums):
    result = []
    path = []

    def backtrack(start):
        result.append(path[:])

        for i in range(start, len(nums)):
            path.append(nums[i])
            backtrack(i + 1)
            path.pop()

    backtrack(0)
    return result
```

The `start` index prevents us from reusing previous elements.

For **Combinations**, we choose `k` numbers from `1` to `n`.

```python
def combine(n, k):
    result = []
    path = []

    def backtrack(start):
        if len(path) == k:
            result.append(path[:])
            return

        for num in range(start, n + 1):
            path.append(num)
            backtrack(num + 1)
            path.pop()

    backtrack(1)
    return result
```

The key idea is:

**Combinations use start index to avoid duplicate ordering.**

**(PPT: Subsets + Combinations)**

## PPT 6 — Permutations

Permutations are different because order matters.

For example:

```text
[1, 2, 3]
```

and

```text
[2, 1, 3]
```

are different permutations.

So instead of using only a start index, we use a `used` array.

```python
def permute(nums):
    result = []
    path = []
    used = [False] * len(nums)

    def backtrack():
        if len(path) == len(nums):
            result.append(path[:])
            return

        for i in range(len(nums)):
            if used[i]:
                continue

            used[i] = True
            path.append(nums[i])

            backtrack()

            path.pop()
            used[i] = False

    backtrack()
    return result
```

At each recursion level, every unused number is a valid next choice.

The key idea is:

**Combinations avoid order. Permutations care about order.**

**(PPT: Permutations — Used Array)**

## PPT 7 — Combination Sum

Combination Sum is another classic backtracking problem.

We are given candidate numbers and a target.

We need all combinations whose sum equals the target.

Each number can be reused.

```python
def combination_sum(candidates, target):
    result = []
    path = []

    def backtrack(start, remaining):
        if remaining == 0:
            result.append(path[:])
            return

        if remaining < 0:
            return

        for i in range(start, len(candidates)):
            path.append(candidates[i])
            backtrack(i, remaining - candidates[i])
            path.pop()

    backtrack(0, target)
    return result
```

Notice this line:

```python
backtrack(i, remaining - candidates[i])
```

We pass `i`, not `i + 1`, because the same candidate can be reused.

If each number can only be used once, we would call `backtrack(i + 1, ...)`.

The key idea is:

**The next start index controls whether a choice can be reused.**

**(PPT: Combination Sum — Reuse vs Use Once)**

## PPT 8 — Handling Duplicates

Duplicate handling is one of the most common backtracking mistakes.

The usual strategy is:

```text
Sort first
Skip duplicate choices at the same recursion level
```

For subsets or combinations with duplicates, the pattern is:

```python
if i > start and nums[i] == nums[i - 1]:
    continue
```

The condition `i > start` is important.

It means we only skip duplicates at the same recursion level.

For permutations with duplicates, the pattern is different:

```python
if i > 0 and nums[i] == nums[i - 1] and not used[i - 1]:
    continue
```

This enforces a consistent order among equal values.

The key idea is:

**Sort first, then skip duplicates only in the correct context.**

**(PPT: Duplicate Handling)**

## PPT 9 — Pruning and Constraint Problems

Backtracking can be expensive because it often explores an exponential search space.

So pruning is critical.

Pruning means stopping a branch early when it cannot lead to a valid solution.

Examples include:

```text
remaining sum < 0
candidate > remaining
invalid parentheses count
queen conflict
path too long
duplicate choice
```

For **Generate Parentheses**, we prune using two rules:

```text
Add "(" only if open_count < n
Add ")" only if close_count < open_count
```

For **N-Queens**, we prune by checking columns and diagonals.

For cell `(row, col)`:

```text
main diagonal = row - col
anti diagonal = row + col
```

The key idea is:

**Pruning cuts branches without losing valid answers.**

**(PPT: Pruning + Constraint Backtracking)**

## PPT 10 — Senior-Level Explanation and Closing Insight

At senior level, I would explain backtracking like this.

Backtracking is a DFS-based technique for exploring a decision tree.

At each recursive call, we maintain a partial solution.

We choose one option, recursively explore deeper, and then undo the choice before trying the next option.

This is the pattern:

```text
choose
explore
unchoose
```

For subsets and combinations, one recursion level usually means choosing the next element from a start index.

For permutations, order matters, so we use a `used` array.

For combination sum, the recursive start index controls whether a candidate can be reused.

For duplicate inputs, we usually sort first and skip duplicates carefully.

For performance, we use pruning to avoid branches that cannot become valid solutions.

The complexity depends on the size of the decision tree.

Subsets are usually **O(2^n)**.

Permutations are **O(n!)**.

Combinations are **O(C(n, k))**.

The final insight is:

**Backtracking is not just brute-force recursion.
It is structured search over choices with state restoration.**

If you can explain the state, the choices, the base case, the pruning rule, duplicate handling, and the undo step, then you understand backtracking at an interview-ready level.

Thank you.

**(PPT: Closing Insight — Decision Tree + Restore State)**
