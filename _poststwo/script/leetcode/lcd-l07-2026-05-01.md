## PPT 1 — Title Slide

Hello everyone.
Today I’d like to walk through a very important stack pattern: **Monotonic Stack**.

A monotonic stack is a stack that keeps elements in increasing or decreasing order.

It is commonly used when we need to find the nearest greater or nearest smaller element around each position.

A typical monotonic stack flow looks like this:

```text
Maintain stack order
→ Read current element
→ Pop elements that break the order
→ Resolve answers for popped elements
→ Push current index
```

The key idea is:

**Monotonic Stack stores unresolved candidates until a boundary appears.**

## PPT 3 — Why Monotonic Stack Works

A brute-force solution often scans from each index to find the next greater or smaller element.

For example:

```python
for i in range(len(nums)):
    for j in range(i + 1, len(nums)):
        if nums[j] > nums[i]:
            answer[i] = nums[j]
            break
```

This is **O(n²)**.

Monotonic stack avoids repeated scanning.

Instead, we keep unresolved indexes in a stack.

When a new value arrives, it may resolve multiple previous elements.

Those elements get popped, and their answers are filled immediately.

Even though there is a while loop inside the for loop, the total work is still linear.

Why?

Because each element is pushed once and popped once.

The key idea is:

**Monotonic Stack turns repeated scanning into one pass with amortized O(n) time.**

**(PPT: Brute Force O(n²) → Monotonic Stack O(n))**

## PPT 4 — Increasing vs Decreasing Stack

There are two main types.

A **monotonic increasing stack** keeps values increasing from bottom to top.

```text
bottom → top
small → large
```

This is usually useful for finding smaller boundaries, like next smaller element or histogram smaller bars.

A **monotonic decreasing stack** keeps values decreasing from bottom to top.

```text
bottom → top
large → small
```

This is usually useful for finding greater boundaries, like next greater element, daily temperatures, and stock span.

The comparison condition decides what gets popped.

For example:

```text
while stack and current breaks the monotonic order:
    pop
```

The key idea is:

**The stack direction depends on what boundary you are trying to find.**

**(PPT: Increasing Stack vs Decreasing Stack)**

## PPT 5 — What Does the Stack Store?

In monotonic stack problems, we usually store **indexes**, not just values.

Why?

Because indexes let us calculate distance, width, position, and still access the original value.

For example:

```text
nums[index]
distance = current_index - index
width = right_boundary - left_boundary - 1
```

Sometimes values are enough, if the problem only asks for values.

And sometimes we store pairs, like:

```text
(price, span)
(value, count)
```

This is useful for Stock Span or contribution counting.

The key idea is:

**Store indexes when position, distance, or width matters.**

**(PPT: Store Indexes vs Values vs Pairs)**

## PPT 6 — Next Greater and Next Smaller Element

For **Next Greater Element**, we maintain a decreasing stack of unresolved indexes.

```python
def next_greater_element(nums):
    result = [-1] * len(nums)
    stack = []

    for i, num in enumerate(nums):
        while stack and nums[stack[-1]] < num:
            index = stack.pop()
            result[index] = num

        stack.append(i)

    return result
```

When the current value is greater than the value at the stack top, the current value is the next greater element for that index.

For **Next Smaller Element**, the idea is symmetric.

We maintain an increasing stack.

When the current value is smaller than the stack top, it resolves that previous index.

The key idea is:

**Next boundary problems are usually resolved when elements are popped.**

**(PPT: Next Greater + Next Smaller)**

## PPT 7 — Previous Greater and Previous Smaller

Previous boundary problems work slightly differently.

For **Previous Greater Element**, we scan from left to right and maintain a decreasing stack.

Before processing the current element, we remove all values that are smaller than or equal to it.

After that, if the stack is not empty, the stack top is the nearest greater element on the left.

```python
def previous_greater_element(nums):
    result = [-1] * len(nums)
    stack = []

    for i, num in enumerate(nums):
        while stack and nums[stack[-1]] <= num:
            stack.pop()

        if stack:
            result[i] = nums[stack[-1]]

        stack.append(i)

    return result
```

For **Previous Smaller Element**, we use an increasing stack and pop values greater than or equal to the current value.

The key idea is:

**Previous boundary is usually read from stack top before pushing the current index.**

**(PPT: Previous Greater + Previous Smaller)**

## PPT 8 — Daily Temperatures and Circular Array

**Daily Temperatures** is a next greater element problem by distance.

We store indexes of days whose warmer future day has not been found.

When the current temperature is warmer than the stack top day, we pop that index and calculate:

```text
answer = current_index - previous_index
```

For circular arrays, like **Next Greater Element II**, we simulate going around the array by looping twice.

```python
def next_greater_elements_circular(nums):
    n = len(nums)
    result = [-1] * n
    stack = []

    for i in range(2 * n):
        index = i % n

        while stack and nums[stack[-1]] < nums[index]:
            prev_index = stack.pop()
            result[prev_index] = nums[index]

        if i < n:
            stack.append(index)

    return result
```

We only push indexes during the first pass.

The second pass only resolves remaining candidates.

The key idea is:

**Circular monotonic stack uses two passes, but each index is still pushed once.**

**(PPT: Daily Temperatures + Circular Next Greater)**

## PPT 9 — Largest Rectangle in Histogram

A classic advanced monotonic stack problem is **Largest Rectangle in Histogram**.

For each bar, we need to know:

```text
first smaller bar on the left
first smaller bar on the right
```

Then:

```text
width = right_smaller_index - left_smaller_index - 1
area = height * width
```

We use a monotonic increasing stack.

When the current height is smaller than the stack top height, the popped bar has found its right smaller boundary.

The new stack top becomes its left smaller boundary.

A sentinel zero is often added at the end to force all remaining bars to be processed.

The key idea is:

**When a bar is popped, we finally know the widest rectangle where that bar is the limiting height.**

**(PPT: Largest Rectangle in Histogram)**

## PPT 10 — Senior-Level Explanation and Closing Insight

At senior level, I would explain monotonic stack like this.

A monotonic stack is a stack that maintains increasing or decreasing order.

It is mainly used for nearest greater or nearest smaller boundary problems.

The stack usually stores unresolved indexes.

When the current element breaks the monotonic invariant, we pop elements and resolve their answers.

For next greater element, we usually use a decreasing stack.

For next smaller element, we usually use an increasing stack.

For previous greater or previous smaller, we pop invalid candidates first, and then the stack top becomes the nearest valid previous boundary.

A critical detail is duplicate handling.

The difference between strict and non-strict comparison matters.

For contribution-counting problems, like sum of subarray minimums, we often use strict comparison on one side and non-strict comparison on the other side to avoid double counting.

The final insight is:

**Monotonic Stack is not just a next-greater template.
It is a way to maintain ordered unresolved candidates and resolve boundaries in one pass.**

If you can explain what the stack stores, whether it is increasing or decreasing, what condition triggers popping, what answer is resolved when popping, and why the total complexity is amortized O(n), then you understand monotonic stack at an interview-ready level.

Thank you.

**(PPT: Closing Insight — Ordered Unresolved Candidates)**
