Prefix sum is a preprocessing technique used to make range-sum and subarray-sum problems faster.

The basic idea is:

Instead of calculating the sum of a range again and again, we precompute cumulative sums once.

## PPT 3 — What Problem Are We Solving?

Prefix sum is useful when we need to answer repeated range-sum or subarray-sum questions.

Common examples include:

Range sum queries.
Subarray sum equals target.
Counting subarrays with a certain sum.
Finding a pivot index.
Difference array problems.
And 2D matrix region sum.

A brute-force range sum looks like this:

```python
total = 0

for i in range(left, right + 1):
    total += nums[i]
```

This takes **O(n)** per query.

If we have many queries, that becomes expensive.

Prefix sum avoids repeated summation by storing cumulative sums ahead of time.

The key idea is:

**Precompute once, then answer range queries quickly.**

**(PPT: Brute Force Range Sum → Prefix Sum)**

## PPT 4 — One-Dimensional Prefix Sum

The standard prefix sum array usually has length `n + 1`.

```python
def build_prefix(nums):
    prefix = [0] * (len(nums) + 1)

    for i in range(len(nums)):
        prefix[i + 1] = prefix[i] + nums[i]

    return prefix
```

Here, `prefix[i]` means:

```text
sum of elements before index i
```

So:

```text
prefix[0] = 0
prefix[1] = nums[0]
prefix[2] = nums[0] + nums[1]
```

Then the sum from `left` to `right` inclusive is:

```python
def range_sum(prefix, left, right):
    return prefix[right + 1] - prefix[left]
```

Using an extra leading zero avoids special cases when the range starts at index zero.

The key idea is:

**Define prefix[i] as the sum before index i, then every range uses the same formula.**

**(PPT: Prefix Array Formula)**

## PPT 5 — Range Sum Query Example

Suppose we have:

```text
nums = [2, 4, 6, 8]
```

The prefix array is:

```text
prefix = [0, 2, 6, 12, 20]
```

If we want the sum from index 1 to index 3:

```text
nums[1] + nums[2] + nums[3]
= 4 + 6 + 8
= 18
```

Using prefix sum:

```text
prefix[4] - prefix[1]
= 20 - 2
= 18
```

For an immutable range sum query class, the code is:

```python
class NumArray:

    def __init__(self, nums):
        self.prefix = [0] * (len(nums) + 1)

        for i in range(len(nums)):
            self.prefix[i + 1] = self.prefix[i] + nums[i]

    def sumRange(self, left, right):
        return self.prefix[right + 1] - self.prefix[left]
```

Build time is **O(n)**.
Each query is **O(1)**.
Space is **O(n)**.

**(PPT: Range Sum Query — O(1) Query)**

## PPT 6 — Prefix Sum With Hash Map

Prefix sum becomes even more powerful when combined with a hash map.

For a subarray from `i` to `j`:

```text
sum(i, j) = prefix[j + 1] - prefix[i]
```

If we want the subarray sum to equal `k`, then:

```text
prefix[j + 1] - prefix[i] = k
```

Rearrange it:

```text
prefix[i] = prefix[j + 1] - k
```

So at each index, if the current prefix sum is `current_sum`, we ask:

```text
Have we seen current_sum - k before?
```

This is why prefix sum often pairs with a hash map.

The hash map stores:

```text
key: prefix sum value
value: how many times this prefix sum appeared
```

The key idea is:

**Subarray sum can be modeled as the difference between current prefix and previous prefix.**

**(PPT: current_sum - previous_sum = k)**

## PPT 7 — Subarray Sum Equals K

A classic problem is:

Given an integer array and integer `k`, return the number of subarrays whose sum equals `k`.

Example:

```text
nums = [1, 1, 1], k = 2
answer = 2
```

The code is:

```python
def subarray_sum(nums, k):
    count = {0: 1}
    current_sum = 0
    result = 0

    for num in nums:
        current_sum += num

        need = current_sum - k

        if need in count:
            result += count[need]

        count[current_sum] = count.get(current_sum, 0) + 1

    return result
```

We initialize the map with:

```python
count = {0: 1}
```

This handles subarrays that start at index zero.

For example, if `nums = [3]` and `k = 3`, then `current_sum` is 3, and we need a previous prefix sum of 0.

The key idea is:

**Every previous prefix sum equal to current_sum - k forms a valid subarray ending here.**

**(PPT: Subarray Sum Equals K)**

## PPT 8 — Why Sliding Window May Not Work

A very important interview point is knowing when not to use sliding window.

For subarray sum equals k, the array may contain negative numbers.

Example:

```text
nums = [1, -1, 1], k = 1
```

Sliding window relies on monotonic behavior:

```text
expand right → sum increases
shrink left → sum decreases
```

But negative numbers break this assumption.

Expanding the window may decrease the sum.
Shrinking the window may increase the sum.

Prefix sum does not rely on monotonic movement.

It only relies on this equation:

```text
current_sum - previous_sum = subarray_sum
```

The key idea is:

**Sliding window needs monotonic behavior. Prefix sum only needs state differences.**

**(PPT: Prefix Sum vs Sliding Window)**

## PPT 9 — 2D Prefix Sum and Difference Array

Prefix sum also extends to two important patterns: **2D prefix sum** and **difference array**.

For 2D prefix sum, we use an extra row and column of zeros.

Each cell stores the sum of the rectangle from the top-left corner to that cell.

The query uses inclusion-exclusion:

```text
region sum
= bottom_right
- above
- left
+ overlap
```

This lets us answer matrix region sum queries in **O(1)** after **O(mn)** preprocessing.

Difference array is the reverse idea.

Prefix sum is good for fast range queries.

Difference array is good for fast range updates.

For example, to add a value to a range:

```text
diff[left] += val
diff[right + 1] -= val
```

Then we use one prefix sum pass to reconstruct the final array.

The key idea is:

**Prefix sum answers range queries. Difference array applies range updates.**

**(PPT: 2D Prefix Sum + Difference Array)**

## PPT 10 — Senior-Level Explanation and Closing Insight

At senior level, I would explain prefix sum like this.

Prefix sum is a preprocessing technique that stores cumulative state.

For one-dimensional arrays, I usually define `prefix[i]` as the sum of elements before index `i`.

Then any inclusive range sum from `left` to `right` becomes:

```text
prefix[right + 1] - prefix[left]
```

This gives **O(1)** range queries after **O(n)** preprocessing.

For subarray counting problems, I use a running prefix sum and a hash map of previous prefix sums.

At each index, I check whether `current_sum - target` has appeared before.

That tells me how many subarrays ending at the current index have the target sum.

Prefix sum is also useful for modulo problems, 2D matrix region sum, and difference array reasoning.

The main mistakes are off-by-one errors, forgetting the initial zero prefix, updating the hash map in the wrong order, and using sliding window when negative numbers break monotonicity.

The final insight is:

**Prefix Sum is not just a cumulative array.
It is a way to convert range problems into differences between prefix states.**

If you can clearly define what `prefix[i]` means, the formula, edge cases, and trade-offs become much easier to explain.

Thank you.

**(PPT: Closing Insight — Prefix State Difference)**
