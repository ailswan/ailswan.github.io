## PPT 1 — Title Slide

Hello everyone.
Today I’d like to walk through another very important LeetCode pattern: **Sliding Window**.

Sliding window is a technique for solving problems involving **contiguous subarrays** or **substrings**.

The basic idea is simple:

Instead of recomputing every possible subarray from scratch, we maintain a moving window and update its state incrementally.

A typical sliding window flow looks like this:

```text
Initialize left pointer
→ Move right pointer to expand the window
→ Update window state
→ Shrink from the left when needed
→ Update the answer
```

The key idea is:

**Sliding window is about maintaining a valid window efficiently.**

**(PPT: Title Slide — Sliding Window)**

## PPT 3 — What Problem Are We Solving?

Sliding window is commonly used for questions like:

Longest substring.
Shortest subarray.
Maximum sum of a fixed-size subarray.
Minimum window containing required characters.
Number of valid subarrays.
Or longest window satisfying a constraint.

A brute-force solution often checks all subarrays:

```text
for left in range(n):
    for right in range(left, n):
        check nums[left:right]
```

This is usually **O(n²)**.

Sliding window improves this by avoiding repeated work.

When the right pointer moves, we add one new element.

When the left pointer moves, we remove one old element.

So each element usually enters and leaves the window at most once.

The key idea is:

**Sliding window turns repeated subarray computation into incremental updates.**

**(PPT: Brute Force O(n²) → Sliding Window O(n))**

## PPT 4 — Fixed-Size vs Variable-Size Window

There are two major types of sliding window problems.

The first type is **fixed-size window**.

This is used when the problem says something like:

```text
subarray of size k
substring of length k
exactly k elements
```

For example, maximum sum of a subarray of size k.

The second type is **variable-size window**.

This is used when the window size depends on a condition.

For example:

Longest substring without repeating characters.
Minimum size subarray sum.
Minimum window substring.
Longest substring with at most k distinct characters.

For fixed-size windows, the window always has size k.

For variable-size windows, the right pointer expands the window, and the left pointer shrinks it when the window becomes invalid.

The key idea is:

**Fixed-size window is controlled by length. Variable-size window is controlled by validity.**

**(PPT: Fixed Window vs Variable Window)**

## PPT 5 — Fixed-Size Window Example

Let’s start with a fixed-size window.

Suppose we want the maximum sum of a subarray of size k.

For example:

```text
nums = [1, 2, 3, 4, 5], k = 3
answer = 12
```

Because `[3, 4, 5]` has the largest sum.

The code looks like this:

```python
def max_sum_subarray(nums, k):
    window_sum = 0
    best = float("-inf")

    for right in range(len(nums)):
        window_sum += nums[right]

        if right >= k - 1:
            best = max(best, window_sum)
            window_sum -= nums[right - k + 1]

    return best
```

The logic is:

Add the right element.
When the window reaches size k, update the answer.
Then remove the leftmost element and continue sliding.

The key idea is:

**For fixed-size windows, add one element and remove one element to keep the window size constant.**

**(PPT: Fixed-Size Window — Max Sum Subarray)**

## PPT 6 — Variable-Size Window Template

Now let’s look at variable-size sliding window.

This is the general template:

```python
left = 0

for right in range(len(nums)):
    add(nums[right])

    while window_is_invalid():
        remove(nums[left])
        left += 1

    update_answer()
```

Every variable-size sliding window needs five things.

A left pointer.
A right pointer.
A window state.
A validity condition.
And an answer update rule.

The right pointer usually expands the window.

If the window becomes invalid, the left pointer shrinks the window until it becomes valid again.

The key idea is:

**Variable-size sliding window is driven by a validity condition.**

**(PPT: Variable-Size Window Template)**

## PPT 7 — Longest Substring Without Repeating Characters

A classic variable-size example is:

Find the length of the longest substring without repeating characters.

For example:

```text
s = "abcabcbb"
answer = 3
```

Because `"abc"` is the longest substring without duplicates.

The code is:

```python
def length_of_longest_substring(s):
    seen = set()
    left = 0
    best = 0

    for right in range(len(s)):
        while s[right] in seen:
            seen.remove(s[left])
            left += 1

        seen.add(s[right])
        best = max(best, right - left + 1)

    return best
```

The invariant is:

```text
The current window has no duplicate characters.
```

When the right pointer sees a duplicate, we move the left pointer until the duplicate is removed.

Then we update the maximum length.

The key idea is:

**Define the invariant first. Then the pointer movement becomes clear.**

**(PPT: Longest Substring — No Duplicate Window)**

## PPT 8 — Minimum Size Subarray Sum

Another important example is minimum size subarray sum.

Given positive integers, find the shortest subarray whose sum is at least target.

Example:

```text
nums = [2, 3, 1, 2, 4, 3], target = 7
answer = 2
```

Because `[4, 3]` has sum 7.

The code is:

```python
def min_subarray_len(target, nums):
    left = 0
    window_sum = 0
    best = float("inf")

    for right in range(len(nums)):
        window_sum += nums[right]

        while window_sum >= target:
            best = min(best, right - left + 1)
            window_sum -= nums[left]
            left += 1

    return 0 if best == float("inf") else best
```

Here, we expand until the sum reaches the target.

Then we shrink from the left to find the smallest valid window.

This works because all numbers are positive.

If negative numbers exist, expanding does not always increase the sum, and shrinking does not always decrease it.

The key idea is:

**Sliding window often depends on monotonic behavior.**

**(PPT: Minimum Size Subarray Sum)**

## PPT 9 — Window State and Common Mistakes

The most important design decision is window state.

For sum problems, the state may be a running sum.

For duplicate problems, it may be a set.

For distinct-count problems, it is usually a frequency map.

For minimum or maximum inside a window, we may need a monotonic deque.

A useful question is:

```text
What information do I need about the current window
to decide whether it is valid?
```

Common mistakes include:

Using sliding window when the condition is not monotonic.
Updating the answer at the wrong time.
Forgetting to remove zero-count keys from a frequency map.
Confusing fixed-size and variable-size windows.
Or forgetting to move the left pointer inside the shrink loop.

The key idea is:

**A sliding window solution is only as good as its window state and invariant.**

**(PPT: Window State + Common Mistakes)**

## PPT 10 — Senior-Level Explanation and Closing Insight

At senior level, I would explain sliding window like this.

Sliding window is a two-pointer technique used for contiguous subarray or substring problems.

The left and right pointers define the current window.

The right pointer expands the window by adding new elements.

The left pointer shrinks the window when the current window violates a condition.

The system maintains window state incrementally, such as a sum, set, frequency map, or count.

This avoids recomputing every possible subarray or substring from scratch.

Most sliding window solutions are **O(n)** because each element enters the window once and leaves the window once.

Space complexity depends on the state.

It can be **O(1)** for a running sum, or **O(k)** / **O(n)** for a hash map or set.

The final insight is:

**Sliding Window is not just moving left and right pointers.
It is about maintaining a dynamic window invariant.**

If you can explain what the window represents, what makes it valid, when to expand, when to shrink, and why each element is processed at most twice, then you understand the pattern at an interview-ready level.

Thank you.

**(PPT: Closing Insight — Window + State + Invariant)**
