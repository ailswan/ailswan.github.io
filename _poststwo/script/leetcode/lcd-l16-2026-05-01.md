Hello everyone.
Today I’d like to walk through another important LeetCode pattern: **Greedy Algorithms**.

A greedy algorithm makes the best local decision at each step, and commits to that decision.

The basic idea is:

```text
Make the best immediate choice
→ Move forward
→ Do not revisit the decision
→ Still reach a globally optimal answer
```

Greedy appears in interval scheduling, Jump Game, Gas Station, Assign Cookies, Partition Labels, Task Scheduler, Stock Buy and Sell, and many resource assignment problems.

The key idea is:

**Greedy works only when a local best choice is globally safe.**

## PPT 3 — Greedy vs Dynamic Programming

Greedy and dynamic programming both solve optimization problems.

But they handle choices differently.

Greedy commits to one local decision.

```text
choose best local option
move on
never revisit
```

Dynamic programming compares multiple possibilities.

```text
try choices
store subproblem results
reuse answers
```

Greedy is better when the local choice can be proven safe.

DP is better when choices interact and we need to compare future outcomes.

For example, if taking the best-looking option now may block a better option later, greedy is dangerous.

The key idea is:

**Greedy commits. DP compares.**

**(PPT: Greedy vs DP)**

## PPT 4 — Greedy Choice Property and Exchange Argument

A problem has the **greedy choice property** if there is always an optimal solution that includes the greedy choice.

That means we can safely make the greedy choice first and reduce the problem.

A common proof is called an **exchange argument**.

The idea is:

```text
Take an optimal solution.
If it does not use the greedy choice,
replace part of it with the greedy choice.
Show the result is not worse.
```

For interval scheduling, greedy chooses the interval that ends earliest.

If an optimal solution chooses a later-ending interval first, we can replace it with the earlier-ending interval.

This leaves at least as much room for future intervals.

The key idea is:

**A greedy proof shows the local choice can be part of an optimal solution.**

**(PPT: Greedy Choice Property + Exchange Argument)**

## PPT 5 — Sorting-Based Greedy and Assign Cookies

Many greedy problems start with sorting.

The sorting key exposes the safe local choice.

For interval scheduling, we sort by end time.

For Assign Cookies, we sort children by greed factor and cookies by size.

The goal is to satisfy as many children as possible.

```python
def find_content_children(g, s):
    g.sort()
    s.sort()

    child = 0
    cookie = 0

    while child < len(g) and cookie < len(s):
        if s[cookie] >= g[child]:
            child += 1

        cookie += 1

    return child
```

The greedy idea is:

Use the smallest cookie that can satisfy the least greedy child.

If a cookie is too small for the current child, it cannot satisfy any greedier child either, so we discard it.

The key idea is:

**Sorting helps match the smallest sufficient resource to the smallest need.**

**(PPT: Sorting Greedy — Assign Cookies)**

## PPT 6 — Jump Game and Jump Game II

Jump Game is a reachability greedy problem.

We track the farthest index reachable so far.

```python
def can_jump(nums):
    farthest = 0

    for i, jump in enumerate(nums):
        if i > farthest:
            return False

        farthest = max(farthest, i + jump)

    return True
```

If we ever reach an index greater than `farthest`, that index is unreachable.

Jump Game II asks for the minimum number of jumps.

The greedy idea is to treat each jump as covering a range.

```python
def jump(nums):
    jumps = 0
    current_end = 0
    farthest = 0

    for i in range(len(nums) - 1):
        farthest = max(farthest, i + nums[i])

        if i == current_end:
            jumps += 1
            current_end = farthest

    return jumps
```

The key idea is:

**For reachability greedy, maintain the farthest safe boundary.**

**(PPT: Jump Game — Farthest Reach)**

## PPT 7 — Gas Station

Gas Station is another classic greedy problem.

We have gas stations in a circle.

If total gas is less than total cost, no solution exists.

```text
total gas < total cost → impossible
```

If total gas is enough, we scan once.

```python
def can_complete_circuit(gas, cost):
    if sum(gas) < sum(cost):
        return -1

    start = 0
    tank = 0

    for i in range(len(gas)):
        tank += gas[i] - cost[i]

        if tank < 0:
            start = i + 1
            tank = 0

    return start
```

The important insight is:

If tank becomes negative at station `i`, then every station from the current start to `i` cannot be a valid start.

So we reset the start to `i + 1`.

The key idea is:

**A failed segment can be safely skipped.**

**(PPT: Gas Station — Reset Start)**

## PPT 8 — Interval Greedy

Interval problems are one of the most common greedy categories.

For selecting the maximum number of non-overlapping intervals, we sort by end time.

```python
def max_non_overlapping(intervals):
    intervals.sort(key=lambda x: x[1])

    count = 0
    end = float("-inf")

    for start, finish in intervals:
        if start >= end:
            count += 1
            end = finish

    return count
```

Why end time?

Because choosing the earliest ending interval preserves the most future options.

For Non-overlapping Intervals, we keep the maximum number of intervals and remove the rest.

For Minimum Arrows to Burst Balloons, we also sort by end and shoot an arrow at the earliest ending balloon’s end.

The key idea is:

**For interval selection, sorting by end time preserves future flexibility.**

**(PPT: Interval Scheduling + Minimum Arrows)**

## PPT 9 — Heap-Based Greedy and Common Mistakes

Sometimes the best greedy choice changes dynamically.

That is when a heap is useful.

A heap helps us repeatedly choose the current best candidate.

Examples include:

```text
highest remaining frequency
earliest ending meeting
cheapest available option
highest profit among available projects
```

Task Scheduler and Reorganize String are common heap-style greedy problems.

But greedy has common mistakes.

The biggest mistake is assuming greedy works without proof.

Another common mistake is using the wrong sorting key.

For example, Merge Intervals usually sorts by start time.

But interval selection usually sorts by end time.

The key idea is:

**Greedy depends on the right choice rule, not just sorting or using a heap.**

## PPT 10 — Senior-Level Explanation and Closing Insight

At senior level, I would explain greedy like this.

A greedy algorithm makes the locally best decision at each step and commits to it.

It is correct only when that local decision can be proven safe.

The key property is the greedy choice property.

That means there exists an optimal solution containing the greedy choice.

A common proof method is an exchange argument.

If an optimal solution uses a different choice, we show that replacing it with the greedy choice does not make the answer worse.

Many greedy problems use sorting.

For intervals, sorting by end time often works because finishing earlier leaves more room for future intervals.

Some greedy problems use reachability, like Jump Game.

Some use global feasibility checks, like Gas Station.

Some use heaps when the best available choice changes over time.

The final insight is:

**Greedy is not just choosing what looks best right now.
It is choosing what can be proven safe.**

If you can explain the greedy choice, the invariant, why the choice preserves future options, and why a counterexample cannot break it, then you understand greedy algorithms at an interview-ready level.

Thank you.

**(PPT: Closing Insight — Safe Local Decision)**
