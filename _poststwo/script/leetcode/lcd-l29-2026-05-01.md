## PPT 3 — Why Not Compare Strings Directly?

Direct substring comparison can be expensive.

For example:

```python
s[i:i+k] == s[j:j+k]
```

This may take:

```text
O(k)
```

time because we compare characters one by one.

If we compare many substrings, the total cost can become large.

For example:

```text
n substrings
each length k
```

can become:

```text
O(n * k)
```

Rolling hash improves this.

After preprocessing, we can compute the hash of any substring in **O(1)** time.

The key idea is:

**Direct comparison costs O(length), while rolling hash can compare substring fingerprints in O(1).**

**(PPT: Direct Comparison vs Hash Comparison)**

## PPT 4 — Polynomial Rolling Hash

The most common rolling hash is polynomial rolling hash.

It treats a string like a number in base `B`.

For example, for:

```text
"abc"
```

If we map:

```text
a = 1
b = 2
c = 3
```

then:

```text
hash("abc") = 1 * B^2 + 2 * B + 3
```

Because numbers can become huge, we usually take modulo:

```text
hash = hash % MOD
```

A common choice is a large prime, such as:

```text
10^9 + 7
10^9 + 9
```

The key idea is:

**Polynomial hash represents a string as a base-powered numeric value.**

**(PPT: Polynomial Hash Formula)**

## PPT 5 — Prefix Hash and Powers

To compute substring hashes efficiently, we build prefix hashes.

Define:

```text
prefix[i] = hash of s[0:i]
```

The recurrence is:

```python
prefix[i + 1] = prefix[i] * BASE + value(s[i])
```

We also precompute powers:

```python
power[i] = BASE^i % MOD
```

Why do we need powers?

Because when we remove the prefix before a substring, we need to shift that prefix hash by the substring length.

The code idea is:

```python
def build_prefix_hash(s, base, mod):
    prefix = [0] * (len(s) + 1)
    power = [1] * (len(s) + 1)

    for i, ch in enumerate(s):
        value = ord(ch)
        prefix[i + 1] = (prefix[i] * base + value) % mod
        power[i + 1] = (power[i] * base) % mod

    return prefix, power
```

The key idea is:

**Prefix hash and power array make substring hash queries fast.**

**(PPT: Prefix Hash + Power Array)**

## PPT 6 — O(1) Substring Hash

For substring:

```text
s[l:r]
```

where `r` is exclusive, the hash is:

```python
hash = prefix[r] - prefix[l] * power[r - l]
```

With modulo:

```python
hash = (prefix[r] - prefix[l] * power[r - l]) % MOD
```

Code:

```python
def get_hash(prefix, power, left, right, mod):
    return (
        prefix[right]
        - prefix[left] * power[right - left]
    ) % mod
```

Why does this work?

`prefix[r]` contains the hash of `s[0:r]`.

`prefix[l] * BASE^(r-l)` aligns the hash of `s[0:l]`, so we can subtract it.

The remaining part is the hash of `s[l:r]`.

The key idea is:

**Substring hash requires subtracting the left prefix after shifting it by the substring length.**

**(PPT: O(1) Substring Hash Formula)**

## PPT 7 — Rabin-Karp String Matching

Rabin-Karp uses rolling hash for string matching.

The goal is to find whether a pattern appears in a text.

The idea is:

```text
Compute pattern hash
Compute hash of every text window with same length
Compare hashes
Verify substring if hashes match
```

Why verify?

Because hash collisions are possible.

```python
def rabin_karp(text, pattern):
    if pattern == "":
        return 0

    n = len(text)
    m = len(pattern)

    if m > n:
        return -1

    rh_text = DoubleRollingHash(text)
    rh_pattern = DoubleRollingHash(pattern)

    pattern_hash = rh_pattern.get_hash(0, m)

    for i in range(n - m + 1):
        if rh_text.get_hash(i, i + m) == pattern_hash:
            if text[i:i + m] == pattern:
                return i

    return -1
```

The key idea is:

**Rabin-Karp compares window hashes, then verifies on hash match to handle collisions.**

**(PPT: Rabin-Karp — Pattern Hash vs Window Hash)**

## PPT 8 — Double Hashing and Collision Handling

The biggest caveat of rolling hash is collision.

A collision means:

```text
hash(a) == hash(b)
but a != b
```

If two hashes are different, the strings are definitely different.

But if two hashes are equal, the strings are only probably equal.

To reduce collision risk, we can use double hashing.

That means computing two hashes with two different mod values.

```python
def get_hash_pair(...):
    return (hash1, hash2)
```

In interviews, I would clearly say:

```text
Hash equality suggests equality,
but does not mathematically guarantee equality under modulo hashing.
```

The key idea is:

**Double hashing makes collisions much less likely, but verification is safest when correctness matters.**

**(PPT: Collision + Double Hashing)**

## PPT 9 — Common Applications

Rolling hash is useful in many substring problems.

One classic problem is **Longest Duplicate Substring**.

The pattern is:

```text
Binary search on substring length
+
rolling hash duplicate check
```

For a fixed length `L`, compute hashes of all substrings of length `L`.

If any hash repeats, a duplicate substring probably exists.

Because if a duplicate exists at length `L`, then duplicates also exist for smaller lengths, we can binary search the maximum valid length.

Rolling hash also helps with substring equality queries.

After preprocessing, each query:

```text
Are s[l1:r1] and s[l2:r2] equal?
```

can be answered by comparing hashes.

For palindrome queries, we can build hashes for the original string and reversed string.

Then:

```text
s[l:r] maps to reversed_s[n-r:n-l]
```

The key idea is:

**Rolling Hash is powerful when many substring identity checks are required.**

**(PPT: Longest Duplicate + Equality + Palindrome Queries)**

## PPT 10 — Senior-Level Explanation and Closing Insight

At senior level, I would explain rolling hash like this.

Rolling hash computes numeric fingerprints for substrings.

The most common version is polynomial rolling hash, where a string is treated as a number in some base.

To query substring hashes efficiently, we precompute prefix hashes and powers of the base.

The prefix recurrence is:

```text
prefix[i + 1] = prefix[i] * base + value(s[i])
```

Then substring hash is:

```text
prefix[r] - prefix[l] * power[r - l]
```

This gives **O(1)** substring hash after **O(n)** preprocessing.

Rolling hash is useful for Rabin-Karp matching, duplicate substring detection, substring equality queries, palindrome queries, and binary search over substring length.

The main caveat is hash collision.

Equal hashes do not always guarantee equal strings.

To reduce risk, we can use double hashing, large prime mods, random base, or verify the actual substring when hashes match.

The final insight is:

**Rolling Hash is not just turning strings into numbers.
It is a way to compare many substrings efficiently using prefix hashes and base powers.**

If you can explain the polynomial hash formula, why substring hash needs the power adjustment, how collisions are handled, and when rolling hash is better than KMP, Trie, or sliding window, then you understand rolling hash at an interview-ready level.
