Hello everyone.
Today I’d like to walk through an important string algorithm: **String Matching with KMP**.

String matching asks a simple question:

```text
Does a pattern exist inside a text?
If yes, where does it start?
```

A classic example is:

```text
text = "hello"
pattern = "ll"
answer = 2
```

The brute force solution tries every starting position.

KMP improves this by using information from the pattern itself.

The key idea is:

**KMP avoids rechecking characters by using prefix-suffix overlap inside the pattern.**

## PPT 2 — Core Framework

When I think about KMP, I usually break it down into five parts.

First, brute force string matching.
Second, why brute force repeats work.
Third, prefix and suffix overlap.
Fourth, the LPS array.
And finally, how KMP uses LPS during search.

The most important concept is:

```text
LPS = Longest Proper Prefix which is also Suffix
```

KMP preprocesses the pattern into an LPS array.

Then, when a mismatch happens, KMP does not move the text pointer backward.

Instead, it moves the pattern pointer using the LPS array.

The key idea is:

**KMP turns mismatch recovery into a pattern-pointer jump.**

**(PPT: Core Framework — Brute Force → LPS → KMP Search)**

## PPT 3 — Brute Force String Matching

The brute force approach is straightforward.

Try every possible starting position in the text.

At each position, compare the pattern character by character.

```python
def str_str(text, pattern):
    if pattern == "":
        return 0

    n = len(text)
    m = len(pattern)

    for i in range(n - m + 1):
        j = 0

        while j < m and text[i + j] == pattern[j]:
            j += 1

        if j == m:
            return i

    return -1
```

The time complexity is:

```text
O(n * m)
```

where `n` is the text length and `m` is the pattern length.

The key problem is that brute force forgets useful matched information.

**(PPT: Brute Force — Try Every Start Position)**

## PPT 4 — Why Brute Force Is Inefficient

Consider this example:

```text
text    = "aaaaaaaaab"
pattern = "aaaab"
```

Brute force repeatedly matches several `"a"` characters, then fails near the end.

Then it shifts by one position and compares many of the same characters again.

This is wasteful.

If we already matched:

```text
aaaa
```

and then failed, we should ask:

```text
Is there any prefix of the pattern
that is also a suffix of what we already matched?
```

That overlap tells us how much of the pattern can still be reused.

The key idea is:

**After a mismatch, KMP reuses the valid prefix-suffix overlap instead of restarting from zero.**

**(PPT: Repeated Work → Reuse Pattern Overlap)**

## PPT 5 — Prefix, Suffix, and LPS

A prefix starts at the beginning of a string.

For:

```text
"abab"
```

prefixes include:

```text
"a", "ab", "aba"
```

A suffix ends at the end of a string.

For:

```text
"abab"
```

suffixes include:

```text
"b", "ab", "bab"
```

In KMP, we use **proper prefix** and **proper suffix**.

That means the whole string itself is not allowed.

For `"abab"`, the longest proper prefix that is also a suffix is:

```text
"ab"
```

So the length is `2`.

The LPS array stores this information for every prefix of the pattern.

The key idea is:

**LPS tells us the longest reusable prefix after a mismatch.**

**(PPT: Prefix + Suffix + LPS Meaning)**

## PPT 6 — Build LPS Array

To build the LPS array, we use two pointers.

```text
i = current index being computed
length = current prefix-suffix match length
```

The code is:

```python
def build_lps(pattern):
    lps = [0] * len(pattern)

    length = 0
    i = 1

    while i < len(pattern):
        if pattern[i] == pattern[length]:
            length += 1
            lps[i] = length
            i += 1
        else:
            if length > 0:
                length = lps[length - 1]
            else:
                lps[i] = 0
                i += 1

    return lps
```

If characters match, we extend the current prefix-suffix match.

If they do not match, we fall back using previous LPS values.

The key idea is:

**Building LPS is also pattern matching, but the pattern is being matched against itself.**

**(PPT: Build LPS — Two Pointers)**

## PPT 7 — KMP Search Process

KMP search uses two pointers:

```text
i = pointer in text
j = pointer in pattern
```

If characters match, both pointers move forward.

```text
text[i] == pattern[j]
i += 1
j += 1
```

If `j` reaches the pattern length, we found a match.

The start index is:

```text
i - j
```

If there is a mismatch and `j` is not zero, we do:

```text
j = lps[j - 1]
```

If `j` is already zero, we move the text pointer forward.

The important part is:

**The text pointer never moves backward.**

**(PPT: KMP Search — Text Pointer Never Goes Back)**

## PPT 8 — KMP Code and All Occurrences

Here is the standard KMP first-match template.

```python
def kmp_search(text, pattern):
    if pattern == "":
        return 0

    lps = build_lps(pattern)

    i = 0
    j = 0

    while i < len(text):
        if text[i] == pattern[j]:
            i += 1
            j += 1

            if j == len(pattern):
                return i - j
        else:
            if j > 0:
                j = lps[j - 1]
            else:
                i += 1

    return -1
```

The complexity is:

```text
Time: O(n + m)
Space: O(m)
```

To find all occurrences, after finding a match, we do not stop.

We record the index and continue with:

```python
j = lps[j - 1]
```

This allows overlapping matches.

For example:

```text
text = "aaaa"
pattern = "aa"
answer = [0, 1, 2]
```

The key idea is:

**After a full match, LPS also helps continue searching for overlapping matches.**

**(PPT: KMP Code + Overlapping Matches)**

## PPT 9 — Repeated Substring and Shortest Palindrome

KMP is not only for substring search.

It is also useful for prefix-suffix problems.

For **Repeated Substring Pattern**, we compute LPS for the full string.

Let:

```text
longest = lps[-1]
unit_length = n - longest
```

If `longest > 0` and `n % unit_length == 0`, then the string is made of repeated units.

```python
def repeated_substring_pattern(s):
    lps = build_lps(s)
    longest = lps[-1]
    unit_length = len(s) - longest

    return longest > 0 and len(s) % unit_length == 0
```

For **Shortest Palindrome**, we find the longest palindromic prefix using:

```text
s + "#" + reversed(s)
```

The last LPS value gives the longest palindromic prefix length.

The key idea is:

**LPS is useful whenever the problem is really about prefix-suffix structure.**

**(PPT: KMP Beyond Search — Repeated Pattern + Palindrome Prefix)**

## PPT 10 — Senior-Level Explanation and Closing Insight

At senior level, I would explain KMP like this.

String matching asks whether a pattern appears inside a text.

Brute force tries every starting position and can take **O(n × m)** time.

KMP improves this by preprocessing the pattern into an LPS array.

The LPS array stores the longest proper prefix that is also a suffix for each prefix of the pattern.

During search, KMP uses two pointers.

When characters match, both pointers move forward.

When there is a mismatch, the text pointer does not move backward.

Instead, the pattern pointer jumps to `lps[j - 1]`.

That is why KMP runs in **O(n + m)** time.

KMP is useful for exact substring search, first occurrence, all occurrences, overlapping matches, repeated substring pattern, string periodicity, and prefix-suffix problems.

But KMP is not always the best tool.

For subsequence problems, two pointers are simpler.

For anagram search, sliding window is better.

For many dictionary words, Trie or Aho-Corasick is better.

The final insight is:

**KMP is not about memorizing code.
It is about understanding which matched prefix can still be reused after a mismatch.**

If you can explain what LPS means, how it is built, why `j = lps[j - 1]`, why the text pointer never moves backward, and how overlapping matches work, then you understand KMP at an interview-ready level.

Thank you.

**(PPT: Closing Insight — Prefix-Suffix Reuse)**
