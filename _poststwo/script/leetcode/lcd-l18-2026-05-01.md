Hello everyone.
Today I’d like to walk through another important graph data structure: **Union Find**.

Union Find is also called **Disjoint Set Union**, or DSU.

It is used to manage groups of connected elements.

The basic idea is:

```text
Each element belongs to a set
→ find tells us which set it belongs to
→ union merges two sets
→ connected elements share the same root
```

Union Find is commonly used for connected components, graph connectivity, cycle detection, Number of Provinces, Redundant Connection, Accounts Merge, Number of Islands II, and Kruskal’s algorithm.

The key idea is:

**Union Find dynamically maintains connected components.**

## PPT 2 — Core Framework

When I think about Union Find, I usually break it down into five parts.

First, what problem pattern it solves.
Second, how the parent array represents groups.
Third, the `find` operation.
Fourth, the `union` operation.
And finally, optimizations like path compression and union by rank or size.

The two core operations are:

```text
find(x)  → return the representative root of x
union(a, b) → merge the sets containing a and b
```

If two nodes have the same root, they are connected.

If they have different roots, they are in different components.

The key idea is:

**The root represents the connected component.**

## PPT 3 — Why Union Find Works

Suppose we have five nodes:

```text
0, 1, 2, 3, 4
```

And edges:

```text
0 - 1
1 - 2
3 - 4
```

Then the connected components are:

```text
{0, 1, 2}
{3, 4}
```

Union Find maintains this grouping efficiently.

Each group has a representative root.

For example:

```text
parent[0] = 0
parent[1] = 0
parent[2] = 0
```

This means nodes 0, 1, and 2 belong to the same set.

The key idea is:

**If two nodes have the same representative root, they are in the same component.**

**(PPT: Connected Components + Representative Root)**

## PPT 4 — Parent Array and Find Operation

Union Find usually uses a parent array.

Initially:

```text
parent[i] = i
```

This means every node starts as its own component.

The `find` operation returns the root of a node.

```python
def find(x):
    while parent[x] != x:
        x = parent[x]

    return x
```

If:

```text
find(a) == find(b)
```

then `a` and `b` are connected.

But without optimization, the tree can become tall.

For example:

```text
4 → 3 → 2 → 1 → 0
```

In that case, `find` can become slow.

The key idea is:

**Find follows parent pointers until it reaches the root.**

**(PPT: Parent Array + Basic Find)**

## PPT 5 — Path Compression

Path compression makes `find` much faster.

When we find the root of a node, we update every node on the path to point directly to the root.

```python
def find(x):
    if parent[x] != x:
        parent[x] = find(parent[x])

    return parent[x]
```

Before compression:

```text
4 → 3 → 2 → 1 → 0
```

After calling `find(4)`:

```text
4 → 0
3 → 0
2 → 0
1 → 0
```

The tree becomes flatter over time.

Future `find` calls become much faster.

The key idea is:

**Path compression turns long parent chains into almost direct root access.**

**(PPT: Path Compression — Flatten the Tree)**

## PPT 6 — Union Operation

The `union` operation merges two sets.

First, we find both roots.

Then, if the roots are different, we attach one root to the other.

```python
def union(a, b):
    root_a = find(a)
    root_b = find(b)

    if root_a == root_b:
        return False

    parent[root_b] = root_a
    return True
```

Returning `False` is useful.

It means `a` and `b` were already connected.

In an undirected graph, adding an edge between two already connected nodes creates a cycle.

The key idea is:

**Union only changes the structure when two different components are merged.**

**(PPT: Union Operation — Merge Components)**

## PPT 7 — Union by Rank or Size

If we always attach roots randomly, the tree may still become tall.

So we use **union by rank** or **union by size**.

Union by rank attaches the smaller-rank tree under the larger-rank tree.

Union by size attaches the smaller component under the larger component.

```python
class UnionFind:
    def __init__(self, n):
        self.parent = list(range(n))
        self.size = [1] * n

    def find(self, x):
        if self.parent[x] != x:
            self.parent[x] = self.find(self.parent[x])
        return self.parent[x]

    def union(self, a, b):
        root_a = self.find(a)
        root_b = self.find(b)

        if root_a == root_b:
            return False

        if self.size[root_a] < self.size[root_b]:
            root_a, root_b = root_b, root_a

        self.parent[root_b] = root_a
        self.size[root_a] += self.size[root_b]
        return True
```

The key idea is:

**Path compression flattens trees, and union by size keeps them shallow.**

**(PPT: Union by Rank / Size)**

## PPT 8 — Connected Components and Cycle Detection

Union Find is very useful for counting connected components.

We start with:

```text
components = n
```

Every successful union reduces the count by one.

```python
def count_components(n, edges):
    uf = UnionFind(n)
    components = n

    for a, b in edges:
        if uf.union(a, b):
            components -= 1

    return components
```

Union Find can also detect cycles in an undirected graph.

For each edge, if both endpoints already have the same root, adding that edge creates a cycle.

This is the core idea behind **Redundant Connection**.

The key idea is:

**A failed union often means the edge connects nodes already in the same component.**

**(PPT: Components + Cycle Detection)**

## PPT 9 — Dynamic Connectivity and Common Use Cases

Union Find is especially strong when connections are added over time.

A classic example is **Number of Islands II**.

Each new land cell starts as a new island.

Then we check its four neighbors.

If a neighbor is already land, we union the two cells.

Every successful union reduces the island count by one.

Union Find also works well for **Accounts Merge**.

We can treat each email as a node.

Emails in the same account are unioned together.

After processing all accounts, emails with the same root belong to the same person.

The key idea is:

**Union Find is ideal for merging groups incrementally.**

**(PPT: Dynamic Connectivity — Islands II + Accounts Merge)**

## PPT 10 — Senior-Level Explanation and Closing Insight

At senior level, I would explain Union Find like this.

Union Find is a data structure for managing disjoint sets and answering connectivity questions.

Each node belongs to a component.

Each component has a representative root.

The `find` operation returns the root of a node.

The `union` operation merges two components if their roots are different.

With path compression and union by rank or size, operations are almost constant time.

Technically, the amortized complexity is:

```text
O(α(n))
```

where α is the inverse Ackermann function.

In practice, it behaves like O(1).

Union Find is useful for connected components, undirected cycle detection, Redundant Connection, Number of Provinces, Accounts Merge, dynamic island counting, equality equations, and Kruskal’s minimum spanning tree algorithm.

But it is not a general graph traversal tool.

It does not give shortest paths, actual paths, directed reachability, or topological order.

The final insight is:

**Union Find is not just a parent array.
It is a way to dynamically maintain connected components.**

If you can explain what each node represents, what the root means, when union really merges, how component count changes, why path compression works, and why Union Find is better than DFS or BFS for dynamic connectivity, then you understand this pattern at an interview-ready level.

Thank you.
