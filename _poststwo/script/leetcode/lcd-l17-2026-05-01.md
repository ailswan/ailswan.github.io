## PPT 1 — Title Slide

Hello everyone.
Today I’d like to walk through a classic system design problem: **Design a Payment System**.

A payment system is not just about calling Stripe, Adyen, or another payment provider.

At a deeper level, it is a correctness-first financial workflow system.

The basic flow looks like this:

```text
User checks out
→ Create payment intent
→ Authorize payment
→ Capture payment
→ Write ledger entries
→ Handle webhooks
→ Reconcile provider state
```

The key idea is:

**Payment system design prioritizes correctness, idempotency, auditability, and security over raw latency.**

**(PPT: Title Slide — Design Payment System)**

## PPT 2 — Core Requirements

For functional requirements, the system should let a user pay for an order.

It should support multiple payment methods, such as credit cards, debit cards, wallets, bank transfers, gift cards, or credits.

It should also support authorization, capture, refund, cancellation, payment status tracking, receipts, provider webhooks, and reconciliation.

For non-functional requirements, the most important ones are correctness and security.

We need idempotent APIs, accurate ledger records, auditable transaction history, reliable retry and recovery, and compliance with payment standards.

The key point is:

**In payments, duplicate charges and missing money movements are much more serious than slightly higher latency.**

**(PPT: Functional + Non-Functional Requirements)**

---

## PPT 3 — Core Concepts

The first core concept is **payment intent**.

A payment intent represents the user’s intent to pay for an order.

For example:

```text
order_id = o123
amount = 50 dollars
currency = USD
status = CREATED
```

Then we have **authorization**.

Authorization reserves money on the user’s payment method.

It answers the question:

```text
Can this user pay this amount?
```

After that, we have **capture**.

Capture actually collects the money.

For simple checkout, authorization and capture may happen together.

For ride sharing, food delivery, hotels, or marketplaces, we may authorize first and capture later after the final amount is known.

We also need refunds and chargebacks.

A refund sends money back to the user.

A chargeback is a dispute initiated by the cardholder or bank.

The key idea is:

**Payment is a stateful money movement workflow, not a single API call.**

**(PPT: Payment Intent → Authorization → Capture → Refund / Chargeback)**

## PPT 4 — Main APIs and Data Model

The main APIs would include:

```text
Create payment intent
Authorize payment
Capture payment
Refund payment
Get payment status
```

All mutation APIs must support an idempotency key.

For example, the capture API may receive:

```text
Idempotency-Key: capture-order-o123-v1
```

For the data model, I would separate several tables.

First, a **payment intent** table to track the high-level lifecycle.

Second, a **payment transaction** table to track operations like authorize, capture, refund, and void.

Third, a **ledger entry** table for accounting records.

And fourth, a **provider event** table to store raw webhook events from payment providers.

The key idea is:

**Payment state, provider events, and accounting records should not be mixed into one table.**

**(PPT: APIs + Tables — Intent, Transaction, Ledger, Provider Event)**

## PPT 5 — High-Level Architecture

At a high level, the architecture looks like this:

```text
Client or Order Service
→ Payment API
→ Payment Orchestrator
→ Risk / Fraud Service
→ Payment Provider Adapter
→ External Payment Provider
```

Then asynchronously:

```text
Provider Webhook
→ Webhook Handler
→ Payment State Updater
→ Ledger Service
→ Reconciliation Jobs
```

The **Payment API** validates requests and enforces idempotency.

The **Payment Orchestrator** manages workflow and state transitions.

The **Provider Adapter** isolates provider-specific logic.

The **Ledger Service** writes immutable accounting entries.

The **Webhook Handler** processes asynchronous provider events.

The key idea is:

**Separate orchestration, provider integration, ledger recording, and webhook processing.**

**(PPT: High-Level Architecture Diagram)**

## PPT 6 — Payment State Machine

I would model payment as a state machine.

Common states include:

```text
CREATED
AUTHORIZING
AUTHORIZED
AUTHORIZATION_FAILED
CAPTURING
CAPTURED
CAPTURE_FAILED
CANCELLED
REFUNDING
REFUNDED
CHARGEBACK
```

A successful flow may look like:

```text
CREATED
→ AUTHORIZING
→ AUTHORIZED
→ CAPTURING
→ CAPTURED
```

A failure flow may look like:

```text
CREATED
→ AUTHORIZING
→ AUTHORIZATION_FAILED
```

A refund flow may look like:

```text
CAPTURED
→ REFUNDING
→ REFUNDED
```

The state machine prevents invalid transitions.

For example, capture should only happen after authorization succeeds.

The key idea is:

**Explicit states make retries, partial failures, and recovery much safer.**

**(PPT: Payment State Machine)**

## PPT 7 — Idempotency and Exactly-Once Effect

Idempotency is essential in payment systems.

Clients and services may retry because of network failures, timeouts, server errors, or provider delays.

Without idempotency, the same request may create duplicate charges.

The system should store:

```text
idempotency key
request hash
previous response
operation status
```

If the same key is used again with the same request body, return the previous response.

If the same key is used with a different request body, reject the request.

This applies to authorize, capture, refund, and cancel.

The goal is not exactly-once execution.

In distributed systems, that is very hard.

The goal is:

```text
at-least-once execution
with exactly-once external effect
```

The key idea is:

**Retries are normal. Duplicate money movement is not acceptable.**

**(PPT: Idempotency Key → Request Hash → Stored Response)**

## PPT 8 — Ledger, Webhooks, and Reconciliation

Payment status alone is not enough.

We also need a ledger.

I would use an append-only double-entry ledger.

Every money movement creates debit and credit entries.

The ledger should be immutable, auditable, balanced, and reconcilable.

Then we need webhooks.

Payment providers often send final status asynchronously.

Webhook processing should:

```text
verify provider signature
store raw event
deduplicate event
process idempotently
validate state transition
write ledger if needed
```

But even with webhooks, internal state and provider state may diverge.

For example, the API call may time out, but the provider actually charged the user.

So we need reconciliation jobs.

Reconciliation periodically compares internal records with provider reports and repairs mismatches.

The key idea is:

**Ledger is the accounting truth, webhooks are async signals, and reconciliation is the safety net.**

**(PPT: Ledger + Webhook + Reconciliation)**

## PPT 9 — Security, Fraud, Failure Handling, and Observability

Security and compliance are critical.

We should avoid storing raw card data.

Instead, use tokenized payment methods from payment providers.

Sensitive data should be encrypted in transit and at rest.

Access should be tightly controlled.

Webhook signatures must be verified.

All payment operations should be audited.

Fraud detection should happen before authorization.

The risk system may use signals like device, IP, user history, amount, velocity, billing address, and shipping address.

Failure handling also matters.

Provider calls may timeout.

Webhooks may be delayed or duplicated.

Internal database updates may fail.

So we need retries with backoff, idempotency keys, state machines, outbox pattern, reconciliation jobs, and manual review for unresolved states.

For observability, I would track authorization success rate, capture success rate, refund success rate, provider timeout rate, webhook lag, reconciliation mismatch count, duplicate request count, fraud decline rate, and chargeback rate.

The key idea is:

**A payment system must be observable, recoverable, and auditable end to end.**

**(PPT: Security + Fraud + Observability)**

## PPT 10 — Senior-Level Explanation and Closing Insight

At senior level, I would explain a payment system like this.

A payment system is a correctness-first financial workflow system.

I would start with a payment intent, which represents the user’s intent to pay for an order.

Then I would model the payment lifecycle as a state machine, including created, authorizing, authorized, capturing, captured, refunding, refunded, failed, and chargeback states.

Every mutation API must be idempotent to prevent duplicate charges or duplicate refunds.

I would separate payment intent, payment transactions, provider events, and ledger entries.

The payment state tracks workflow progress.

The append-only double-entry ledger provides accounting truth.

External payment providers should be isolated behind provider adapters because they have different APIs, errors, retry behavior, and webhook formats.

Webhooks must be verified, stored, deduplicated, and processed idempotently.

Reconciliation jobs are required to repair mismatches caused by timeouts, lost webhooks, or partial failures.

The final insight is:

**A payment system is not just calling a payment API.
It is a correctness-first system built around state machines, idempotency, ledger, webhooks, and reconciliation.**

The goal is to move money correctly, prevent duplicate charges, maintain auditable records, and recover safely from partial failures.

Thank you.

**(PPT: Closing Insight — Correctness-First Financial Workflow)**
