Hello everyone.
Today I’d like to walk through another fundamental LeetCode pattern: **DFS Basics**.

DFS stands for **Depth-First Search**.

It is a traversal algorithm that explores one path as deeply as possible before backtracking.

The basic idea is:

```text
Start from a node
→ Go deeper into one branch
→ Stop when we hit a base case
→ Backtrack
→ Explore another branch
```

DFS is commonly used for tree traversal, graph traversal, connected components, path existence, cycle detection, grid traversal, and backtracking.

The key idea is:

**DFS is deep exploration plus backtracking.**

## PPT 3 — Why DFS Works

DFS works because many structures are naturally recursive.

For a tree, each node has children.

For a graph, each node has neighbors.

For a grid, each cell has adjacent cells.

DFS follows one branch until it cannot continue.

It stops when it hits something like:

```text
null
visited node
boundary
obstacle
target
base case
```

Then it returns to the previous call and explores another branch.

For example, in a tree:

```text
A
├── B
│   ├── D
│   └── E
└── C
```

DFS may visit:

```text
A → B → D → E → C
```

The key idea is:

**DFS keeps going deeper until it must return.**

**(PPT: Depth-First Behavior)**

## PPT 4 — DFS vs BFS

DFS and BFS are both traversal algorithms, but they are useful in different situations.

DFS uses recursion or a stack.

It explores deeply first.

DFS is usually good for:

```text
path existence
connected components
backtracking
cycle detection
tree recursion
topological exploration
```

BFS uses a queue.

It explores level by level.

BFS is usually better for:

```text
shortest path in unweighted graphs
minimum steps
level-order traversal
nearest target
multi-source expansion
```

So if the problem asks for minimum steps, BFS is often better.

But if the problem asks to explore all possible paths, detect components, or use recursive structure, DFS is usually more natural.

The key idea is:

**DFS explores depth. BFS explores distance.**

## PPT 5 — Recursive and Iterative DFS Templates

The recursive DFS template is very simple:

```python
def dfs(node, graph, visited):
    if node in visited:
        return

    visited.add(node)

    for neighbor in graph[node]:
        dfs(neighbor, graph, visited)
```

The key components are:

Current node.
Base case.
Visited set if cycles are possible.
Neighbor generation.
Recursive calls.

DFS can also be written iteratively:

```python
def dfs_iterative(start, graph):
    stack = [start]
    visited = set([start])
    order = []

    while stack:
        node = stack.pop()
        order.append(node)

        for neighbor in graph[node]:
            if neighbor not in visited:
                visited.add(neighbor)
                stack.append(neighbor)

    return order
```

The key idea is:

**Recursive DFS uses the call stack. Iterative DFS uses an explicit stack.**

**(PPT: Recursive DFS vs Iterative DFS)**

## PPT 6 — DFS on Binary Trees

DFS is very natural for binary trees.

There are three common traversal orders:

```text
preorder:  root → left → right
inorder:   left → root → right
postorder: left → right → root
```

The difference is when we process the root.

A common DFS tree problem is maximum depth:

```python
def max_depth(root):
    if not root:
        return 0

    left_depth = max_depth(root.left)
    right_depth = max_depth(root.right)

    return 1 + max(left_depth, right_depth)
```

This works because the maximum depth of a node is:

```text
1 + max(left subtree depth, right subtree depth)
```

The key idea is:

**Tree DFS often returns information from children back to the parent.**

**(PPT: Tree DFS — Traversal and Return Values)**

## PPT 7 — Path Sum and DFS State

DFS can also carry state along the current path.

A classic example is **Path Sum**.

The problem asks whether there is a root-to-leaf path whose sum equals a target.

```python
def has_path_sum(root, target_sum):
    if not root:
        return False

    if not root.left and not root.right:
        return target_sum == root.val

    remaining = target_sum - root.val

    return (
        has_path_sum(root.left, remaining)
        or
        has_path_sum(root.right, remaining)
    )
```

Here, the state is the remaining target sum.

At each node, we subtract the current value.

When we reach a leaf, we check whether the remaining sum matches the leaf value.

The key idea is:

**DFS is useful when each recursive call represents one step deeper along a path.**

**(PPT: Path Sum — Carry State Downward)**

## PPT 8 — DFS on Graphs and Connected Components

For graph DFS, a visited set is usually required because graphs can contain cycles.

Without visited, this graph can recurse forever:

```text
A ↔ B
```

The template is:

```python
def graph_dfs(graph, start):
    visited = set()
    order = []

    def dfs(node):
        if node in visited:
            return

        visited.add(node)
        order.append(node)

        for neighbor in graph[node]:
            dfs(neighbor)

    dfs(start)
    return order
```

DFS is also useful for counting connected components.

We iterate through all nodes.

Whenever we find an unvisited node, we start a DFS.

That DFS visits one full component.

The key idea is:

**Each DFS from an unvisited node discovers one connected component.**

**(PPT: Graph DFS + Connected Components)**

## PPT 9 — Grid DFS and Cycle Detection

Grid DFS treats each cell as a graph node.

For example, in **Number of Islands**, each land cell is a node, and neighboring land cells are connected.

When we find a land cell, we count one island and run DFS to mark all connected land as visited.

The base cases are:

```text
out of bounds
not target value
already visited
obstacle
```

DFS is also useful for cycle detection.

In an undirected graph, we pass a parent node.

If we visit an already visited neighbor that is not the parent, we found a cycle.

In a directed graph, we often use three states:

```text
0 = unvisited
1 = visiting
2 = visited
```

If DFS reaches a node that is currently visiting, there is a back edge, so a cycle exists.

The key idea is:

**DFS can track traversal state to detect cycles.**

**(PPT: Grid DFS + Cycle Detection)**

## PPT 10 — Senior-Level Explanation and Closing Insight

At senior level, I would explain DFS like this.

DFS is a traversal algorithm that explores one branch as deeply as possible before backtracking.

It can be implemented recursively using the call stack, or iteratively using an explicit stack.

For trees, DFS is natural because many tree problems can be defined recursively.

For graphs, DFS usually requires a visited set to avoid cycles.

For disconnected graphs, we must start DFS from every unvisited node.

For grids, each cell can be treated as a graph node, and neighbors are generated using directions.

For backtracking, DFS explores a decision tree using the pattern:

```text
choose
explore
unchoose
```

DFS can also return information upward, such as tree height, subtree size, whether a subtree is valid, or whether a target exists.

The complexity is usually **O(n)** for trees, **O(V + E)** for graphs, and **O(mn)** for grids.

The final insight is:

**DFS is not just recursion.
It is deep exploration, base cases, visited state, and backtracking.**

If you can explain what the node or state is, what the base case is, how neighbors are generated, when visited is marked, and whether information should be returned upward, then you understand DFS at an interview-ready level.

Thank you.

**(PPT: Closing Insight — Deep Exploration + Backtracking)**
