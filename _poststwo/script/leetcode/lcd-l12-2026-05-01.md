## PPT 1 — Title Slide

Hello everyone.
Today I’d like to walk through one of the most important algorithm patterns: **Dynamic Programming**.

Dynamic programming, or DP, is used when a problem can be broken into smaller subproblems, and those subproblems repeat.

The basic idea is:

```text
Solve each subproblem once
→ Store the result
→ Reuse it later
→ Avoid repeated computation
```

DP is commonly used for Fibonacci-style problems, climbing stairs, house robber, coin change, knapsack, grid paths, longest common subsequence, edit distance, and many sequence optimization problems.

The key idea is:

**Dynamic Programming turns repeated recursion into reusable subproblem results.**

**(PPT: Title Slide — Dynamic Programming Intro)**

## PPT 2 — Core Framework

When I think about DP, I usually break it down into five parts.

First, what problem pattern DP solves.
Second, overlapping subproblems and optimal substructure.
Third, state definition.
Fourth, recurrence relation and base cases.
And finally, top-down memoization, bottom-up tabulation, and space optimization.

The most important question is:

```text
What does dp[i] mean?
```

If the state definition is unclear, the recurrence usually becomes confusing.

So before writing code, I always try to describe the state in plain English.

For example:

```text
dp[i] = number of ways to reach step i
dp[i] = best answer using the first i elements
dp[i][j] = answer using two prefixes
```

The key idea is:

**Good DP starts with a clear state definition.**

## PPT 3 — Why DP Works

Let’s use Fibonacci as a simple example.

The recurrence is:

```text
fib(n) = fib(n - 1) + fib(n - 2)
```

A naive recursive solution looks simple:

```python
def fib(n):
    if n <= 1:
        return n

    return fib(n - 1) + fib(n - 2)
```

But this recomputes the same values many times.

For example:

```text
fib(5) needs fib(3)
fib(4) also needs fib(3)
```

So `fib(3)` is solved again and again.

DP fixes this by computing each subproblem once and storing the result.

The key idea is:

**DP works when repeated subproblems can be cached and reused.**

**(PPT: Naive Recursion → Repeated Work → DP Cache)**

## PPT 4 — Two Required Conditions

DP usually applies when two conditions exist.

The first condition is **overlapping subproblems**.

This means the same smaller problems appear multiple times.

Fibonacci is the classic example because many recursive branches ask for the same `fib(x)` value.

The second condition is **optimal substructure**.

This means the answer to a larger problem can be built from answers to smaller problems.

For example:

```text
fib(n) = fib(n - 1) + fib(n - 2)
```

For House Robber:

```text
best up to house i
= max(skip house i, rob house i)
```

The key idea is:

**DP needs reusable subproblems and a way to combine smaller answers into a larger answer.**

**(PPT: Overlapping Subproblems + Optimal Substructure)**

## PPT 5 — DP Design Process

A useful DP design process has six questions.

First, what is the state?

Second, what does `dp[i]` or `dp[i][j]` represent?

Third, what is the recurrence?

Fourth, what are the base cases?

Fifth, what is the iteration order?

And sixth, can space be optimized?

The state describes the subproblem.

The recurrence explains how to compute the current state from smaller states.

The base cases initialize the smallest valid subproblems.

The iteration order ensures every dependency is already computed before we use it.

The key idea is:

**DP is a modeling process, not just memorizing formulas.**

**(PPT: Six-Step DP Design Process)**

## PPT 6 — Top-Down DP With Memoization

The first implementation style is **top-down DP**.

Top-down DP starts from the original problem, uses recursion, and caches results.

Here is Fibonacci with memoization:

```python
def fib(n):
    memo = {}

    def dfs(x):
        if x <= 1:
            return x

        if x in memo:
            return memo[x]

        memo[x] = dfs(x - 1) + dfs(x - 2)
        return memo[x]

    return dfs(n)
```

This version follows the natural recursive structure.

The only difference is that every subproblem result is stored.

So when the same state appears again, we return the cached value.

The key idea is:

**Top-down DP is recursion plus memoization.**

**(PPT: Top-Down Memoization)**

---

## PPT 7 — Bottom-Up DP With Tabulation

The second implementation style is **bottom-up DP**.

Bottom-up DP starts from base cases and fills a table iteratively.

Here is Fibonacci with tabulation:

```python
def fib(n):
    if n <= 1:
        return n

    dp = [0] * (n + 1)
    dp[0] = 0
    dp[1] = 1

    for i in range(2, n + 1):
        dp[i] = dp[i - 1] + dp[i - 2]

    return dp[n]
```

This avoids recursion overhead.

But it requires us to know the correct iteration order.

When computing `dp[i]`, both `dp[i - 1]` and `dp[i - 2]` must already be known.

The key idea is:

**Bottom-up DP builds from base cases to the final answer.**

**(PPT: Bottom-Up Tabulation)**

## PPT 8 — Classic Example: Climbing Stairs

A classic DP problem is **Climbing Stairs**.

You can climb either one or two steps.

How many ways are there to reach step `n`?

State definition:

```text
dp[i] = number of ways to reach step i
```

To reach step `i`, the last move must come from step `i - 1` or step `i - 2`.

So the recurrence is:

```text
dp[i] = dp[i - 1] + dp[i - 2]
```

Code:

```python
def climb_stairs(n):
    if n <= 2:
        return n

    dp = [0] * (n + 1)
    dp[1] = 1
    dp[2] = 2

    for i in range(3, n + 1):
        dp[i] = dp[i - 1] + dp[i - 2]

    return dp[n]
```

The key idea is:

**Define the state first, then the recurrence becomes natural.**

**(PPT: Climbing Stairs — State and Recurrence)**

## PPT 9 — Space Optimization and Common Patterns

Sometimes we do not need the full DP table.

For Climbing Stairs, `dp[i]` only depends on two previous states.

So we can use two variables:

```python
def climb_stairs(n):
    if n <= 2:
        return n

    prev2 = 1
    prev1 = 2

    for i in range(3, n + 1):
        current = prev1 + prev2
        prev2 = prev1
        prev1 = current

    return prev1
```

This reduces space from **O(n)** to **O(1)**.

Many DP problems fall into common patterns.

Fibonacci-style DP depends on previous states.

Knapsack DP uses choose-or-skip decisions.

Grid DP depends on neighboring cells.

Two-string DP uses two indexes.

Interval DP solves ranges.

The key idea is:

**Space optimization is safe only after we understand state dependencies.**

**(PPT: Space Optimization + DP Pattern Map)**

## PPT 10 — Senior-Level Explanation and Closing Insight

At senior level, I would explain dynamic programming like this.

Dynamic programming solves problems with overlapping subproblems and optimal substructure.

Instead of recomputing the same subproblem many times, we solve each subproblem once, store its result, and reuse it.

The first step is defining the state.

A state represents a smaller subproblem.

After defining the state, we derive the recurrence relation.

The recurrence tells us how to compute the current state from smaller states.

Then we define base cases and choose an implementation style.

Top-down DP uses recursion plus memoization.

Bottom-up DP uses iteration plus a table.

The complexity is usually:

```text
number of states × transition cost
```

The final insight is:

**Dynamic Programming is not about memorizing formulas.
It is about modeling reusable subproblems.**

If you can explain what the state means, why subproblems overlap, how the recurrence is derived, what the base cases are, and whether space can be optimized safely, then you understand DP at an interview-ready level.

Thank you.

**(PPT: Closing Insight — Reusable Subproblems)**
