Hello everyone.

Design data structure problems ask us to implement a custom class with specific operations.

For example:

```text
get
put
add
remove
search
getRandom
findMedian
```

The challenge is not just making the operations correct.

The real challenge is meeting the required time complexity.

The key idea is:

**Data structure design is about matching operations to the right combination of underlying structures.**

## PPT 2 — Core Framework

When I think about data structure design problems, I usually break them down into five parts.

First, list all required operations.
Second, identify the target complexity for each operation.
Third, choose the right underlying structures.
Fourth, define internal invariants.
And finally, make sure every method keeps those invariants correct.

A useful mapping is:

```text
Need fast lookup?        → HashMap
Need random access?      → Array
Need fast reordering?    → Doubly linked list
Need min or max?         → Heap
Need prefix search?      → Trie
Need timestamp lookup?   → Sorted list + binary search
```

The key idea is:

**Start from operations and complexity, then choose the data structure combination.**

**(PPT: Core Framework — Operations → Complexity → Structures)**

## PPT 3 — Common Building Blocks

Most design problems are solved by combining basic data structures.

A **HashMap** gives fast key lookup.

An **array** gives index access and random selection.

A **linked list** gives O(1) remove and move operations when we already have the node.

A **heap** gives efficient min, max, top k, and median support.

A **Trie** gives prefix search for strings.

For example, LRU Cache is not just a HashMap problem.

It needs:

```text
HashMap for O(1) lookup
Doubly linked list for O(1) recency update
```

The key idea is:

**Most design problems need a combination, not a single structure.**

**(PPT: Building Blocks — HashMap, Array, List, Heap, Trie)**

## PPT 4 — Min Stack and Queue Using Stacks

A simple design problem is **Min Stack**.

It needs to support:

```text
push
pop
top
getMin
```

all in O(1).

The solution uses two stacks.

```text
main stack → stores values
min stack  → stores current minimum at each depth
```

Every time we push a value, we also push the current minimum.

Every time we pop, both stacks pop together.

Another classic problem is **Queue Using Stacks**.

We use two stacks:

```text
input stack
output stack
```

Push goes into input.

Pop comes from output.

If output is empty, we move all input elements to output.

This reverses the order and gives FIFO behavior.

The key idea is:

**Auxiliary structures help simulate missing operations efficiently.**

**(PPT: Min Stack + Queue Using Stacks)**

## PPT 5 — LRU Cache

A very important design problem is **LRU Cache**.

It supports:

```text
get(key)
put(key, value)
```

both in O(1).

When capacity is exceeded, we remove the least recently used item.

To do this, we need two structures:

```text
HashMap: key → linked list node
Doubly linked list: recency order
```

The linked list order is:

```text
front = most recently used
back  = least recently used
```

On every `get`, we move the node to the front.

On every `put`, we insert or update the node and move it to the front.

If capacity is exceeded, we remove the node at the back.

The key idea is:

**LRU Cache combines O(1) lookup with O(1) recency updates.**

**(PPT: LRU Cache — HashMap + Doubly Linked List)**

## PPT 6 — RandomizedSet

Another classic problem is **Insert Delete GetRandom O(1)**.

We need:

```text
insert(val)
remove(val)
getRandom()
```

all average O(1).

The solution uses:

```text
array for random access
HashMap value → index
```

Insert appends to the array and records the index.

GetRandom chooses a random element from the array.

The tricky part is remove.

To remove in O(1), we swap the element with the last element, pop the last element, and update the moved element’s index.

```text
remove target
→ swap with last
→ pop last
→ update index map
```

The key idea is:

**Array gives random access, and HashMap gives O(1) index lookup.**

**(PPT: RandomizedSet — Array + Index Map)**

## PPT 7 — TimeMap and SnapshotArray

Some design problems are about history.

A classic example is **Time-Based Key-Value Store**.

It supports:

```text
set(key, value, timestamp)
get(key, timestamp)
```

For each key, we store a sorted list:

```text
key → [(timestamp, value)]
```

Because timestamps are inserted in increasing order, set is O(1).

For get, we binary search for the largest timestamp less than or equal to the query timestamp.

**SnapshotArray** is similar.

Instead of copying the whole array on every snapshot, we store change history per index:

```text
index → [(snap_id, value)]
```

Then get uses binary search.

The key idea is:

**For historical lookup, store changes and use binary search instead of copying everything.**

**(PPT: TimeMap + SnapshotArray)**

## PPT 8 — MedianFinder and Heap-Based Design

For stream problems, heap-based design is very common.

**MedianFinder** supports:

```text
addNum(num)
findMedian()
```

We use two heaps:

```text
max heap for smaller half
min heap for larger half
```

In Python, we simulate a max heap using negative values.

The invariants are:

```text
all values in small <= all values in large
sizes differ by at most one
```

When a new number is added, we insert it, rebalance the heaps, and keep the two halves valid.

If the total count is odd, the median is the top of the larger half.

If the total count is even, the median is the average of both heap tops.

The key idea is:

**Two heaps split a stream into lower half and upper half.**

**(PPT: MedianFinder — Two Heaps + Invariants)**

## PPT 9 — Trie and Prefix-Based Design

When the problem needs prefix search, Trie is often the right structure.

A Trie supports:

```text
insert(word)
search(word)
startsWith(prefix)
```

Each Trie node stores:

```text
children map
is_word flag
```

Insert walks through characters and creates missing nodes.

Search returns true only if the final node marks a complete word.

startsWith only checks whether the prefix path exists.

For **Word Dictionary**, search may contain a wildcard dot.

When we see a normal character, we follow one child.

When we see `"."`, we try all children with DFS.

The key idea is:

**Trie turns string prefix operations into tree traversal.**

**(PPT: Trie + Word Dictionary Wildcard Search)**

## PPT 10 — Senior-Level Explanation and Closing Insight

At senior level, I would explain design data structure problems like this.

First, list every public operation.

Second, write the required time complexity for each method.

Then map each requirement to an internal structure.

If I need O(1) lookup, I usually need a HashMap.

If I need O(1) random access, I use an array.

If I need O(1) deletion from an array, I combine it with a HashMap and use swap-with-last.

If I need recency ordering, I use a doubly linked list.

If I need median in a stream, I use two heaps.

If I need prefix search, I use a Trie.

The most important part is maintaining invariants.

For LRU, the linked list must always match recency order.

For RandomizedSet, the array and index map must always agree.

For MedianFinder, the two heaps must stay balanced.

The final insight is:

**Design Data Structures is not just knowing HashMap.
It is mapping operation requirements to a combination of structures and preserving invariants after every method call.**

If you can explain the operations, complexity per method, internal structures, invariants, edge cases, and update logic, then you understand data structure design at an interview-ready level.

Thank you.

**(PPT: Closing Insight — Operations, Structures, and Invariants)**
